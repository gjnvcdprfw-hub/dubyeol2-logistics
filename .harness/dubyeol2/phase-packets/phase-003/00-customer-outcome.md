## 1. 연결 맥락

- 연결된 고객 흐름: 흐름 1 (마지막 조각)
- 연결된 메뉴탭 / 화면 영역: 셀러 주문 상세(견적) + 운영자 견적 입력
- 연결된 기능 약속: Feature 1 (이 Phase로 완료)
- 연결된 Phase: Phase 3

## 1.1 플랫폼 독립 원칙 확인

- 고객 계약: 셀러가 주문의 총비용을 항목별로 확인한다 (숨은 항목 없음)
- 입력 방식 후보: 운영자 웹 입력(단가·중국내배송비·무게·부피·환율) + 요율 자동 계산
- 출력 방식 후보: 셀러 주문 상세 내 견적 카드
- 실행 엔진 후보: 무관. 고객 계약에서 제외: 특정 환율 API, 결제사
- 유지되어야 하는 고객 결과: "항목별 투명 견적"

## 2. Phase 고객 목표

- 이번 Phase에서 고객이 끝낼 수 있어야 하는 일: 입고 완료된 주문의 견적 — 상품가(구매대행 건), 중국 내 배송비, 대행 수수료(5%)+수수료 VAT(10%), 검수비(유료 신청 건), 예상 국제운임(해운/항공, 무게·부피 기준), 합계, 적용 환율 — 을 항목별로 확인하고 "국제배송 보낼지" 결정
- 이 일이 안 되면 고객에게 생기는 피해: 총비용을 모른 채 결정 — 숨은비용 불신
- 이번 Phase가 끝나면: Feature 1 완료. 다음 기능(LCL 국제배송)의 입구

## 3. 고객 결과 요청

### 고객 결과 요청 1

- 고객에게 생기는 변화: 운영자가 견적 입력(상품 단가 ¥·중국 내 배송비 ¥·실측 무게 kg·부피 CBM·적용 환율 ₩/¥·배송 방식 해운/항공)을 하면, 시스템이 두리무역 요율로 자동 계산(수수료=상품가 5%, 수수료 VAT=수수료의 10%, 검수비=유료 신청 건만, 예상 운임=방식별 요율×청구중량[실중량 vs 부피중량 큰 값])해 셀러 주문 상세에 **항목별 견적 카드**로 표시
- 완료 기준: feature.md 6번 합격 예시 — "미니가전 셀러 A ×100, 12kg/0.08CBM" 건에서 항목별 금액이 표시되고 계산이 요율표와 일치. 검수 미신청 건은 검수비 항목 자체가 없음(0원 표시 아님)
- 이번에 안 하는 것: 결제·예치금·실청구, 환율 자동 연동, 관세·부가세 계산(별도 안내 문구만), 견적 수정 이력, LCL 요율(해운택배·항공만 — LCL은 다음 기능)
- 고객 피해 방지 기준: 견적 항목 누락·합계 불일치 금지, "예상 운임·참고용, 관세·부가세 별도" 면책 문구(두리무역 패턴), 타 셀러 견적 노출 금지, 요율 값은 코드 상수 아닌 단일 요율 정의 파일(대표님 컷팅 대비)
- 실제 한 건의 QA 예시: 보온병 ×100(검수 신청, 입고완료) — 운영자: 단가 ¥20·중국배송 ¥30·12kg·0.08CBM·환율 190·해운 → 상품가 ¥2,000, 수수료 ¥100(5%), 수수료 VAT ¥10, 검수비 ¥100(¥1×100), 예상 운임: 청구중량 max(12, 0.08×1,000,000÷6000=13.33)→**올림 14kg**(kg 단위 과금) → ¥25+13×¥5=**¥90**, 중국 내 배송 ¥30 — 합계 **¥2,330** = ₩442,700 (환율 190, 셀러 화면은 ₩ 병기)
- 통과해도 실패인 경우: ①합계≠항목 합 ②검수 미신청 건에 검수비 표시 ③요율이 코드 곳곳에 하드코딩(컷팅 불가 구조)

## 4. PM 후보 판단

- PM이 판단하고 진행할 수 있는 것: 견적 카드 구성(두리무역 가격정보 탭 벤치), 검수비 단가는 두리무역 표기 병존(¥35/시간 vs ¥1/개) 중 **¥1/개 채택**(수량 기반이라 자동 계산 가능 — 시간제는 운영 입력 필요, 추후 컷팅 대상)
- 대표님 판단이 필요한 것: 없음 (요율 컷팅은 추후)
- 후보가 너무 클 수 있는 이유: 요율 엔진+운영자 입력+셀러 표시가 한 덩어리 — 단 Phase 2에서 운영자 입력·셀러 표시 패턴이 확립돼 있어 실질 신규는 요율 계산뿐
- 너무 작을 수 있는 이유: 없음
- 절대 묶으면 안 되는 것: LCL·결제·출고(다음 기능)
- Team Leader에게 요청하는 범위 판단: 한 Phase 적정 여부
