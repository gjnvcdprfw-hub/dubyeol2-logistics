# Phase 1 고객 결과 요청

상태: PM 후보 작성 완료 (2026-07-04) → Team Leader 범위 판단 대기

이 파일은 PM이 Team Leader에게 넘기는 후보 고객 결과 요청이다. 구현 티켓이 아니라 고객 결과 단위 작업만 적는다.

PM은 이 파일만으로 Phase 경계를 최종 확정하지 않는다. Team Leader가 `superpowers:brainstorming`으로 범위를 판단한 뒤, PM이 Phase 경계를 확정한다.

## 0. L1~L5 정렬 체크

## 1. 연결 맥락

- 연결된 고객 흐름: 흐름 1 (주문을 맡기고 입고·검수·견적을 확인)
- 연결된 메뉴탭 / 화면 영역: 셀러 주문 접수 · 셀러 대시보드(입고·검수·견적)
- 연결된 기능 약속: Feature 1 (승인 2026-07-04)
- 연결된 Phase: Phase 1

## 1.1 플랫폼 독립 원칙 확인

- 고객 계약: 셀러가 주문을 맡기면 → 안전 입고 → 입고 증거(사진·외포장 안내, 유료 신청 시 수량·외관·하자 검수) → 항목별 투명 견적 확인
- 이번 Phase의 입력 방식 후보: 웹 주문 폼(링크 붙여넣기), (뒤 Phase: 엑셀 업로드·1688 연동)
- 이번 Phase의 출력 방식 후보: 웹 대시보드
- 이번 Phase의 실행 엔진 후보: 무관
- 고객 계약에서 제외할 구현 선택지: 특정 결제사, 특정 DB, 1688 OAuth, AI HS코드, 특정 WMS
- 입력/출력/실행 엔진이 바뀌어도 유지되어야 하는 고객 결과: "안전 입고 + 입고 증거 + 투명 견적" 세 가지 확인

## 2. Phase 고객 목표

- 이번 Phase에서 고객이 끝낼 수 있어야 하는 일: 주문 접수부터 입고 증거·(유료 신청 시)검수 결과·항목별 견적 확인까지 화면에서 스스로 완결
- 이 일이 안 되면 고객에게 생기는 피해: 깜깜이로 맡겨야 해서 분실·하자·숨은비용 리스크를 셀러가 떠안고, 물량을 못 맡김
- 이번 Phase가 끝나면 고객이 다음으로 할 수 있는 일: "이 물량을 국제배송으로 보내겠다" 결정 (Feature 2 진입)

## 3. 고객 결과 요청

### 고객 결과 요청 1

- 고객에게 생기는 변화: 셀러가 (가입·로그인 후) 상품 링크·수량·옵션·구분(구매/배송대행)으로 주문을 접수하고, 유료 검수를 선택할 수 있으며, 운영이 입고·확인을 기록하면 대시보드에서 ①입고 상태 ②입고 사진 1~2장+외포장 이상 안내 ③(유료 신청 건) 수량·외관·하자 검수 결과 ④항목별 견적(상품가·수수료·검수비·예상 국제운임)을 확인한다
- 완료 기준: feature.md 6번 합격 예시 1건이 실제 화면에서 통과. 요율은 두리무역 초기값(benchmark-duly.md §3) 사용, UI는 design-system.md 토큰 준수
- 이번에 안 하는 것: LCL·통관·한국배송·추적, 1688 OAuth·자동 불러오기, AI HS코드, 3PL·WMS 바코드, 실제 1688 매입·결제 자동화(모의/수동), 실서비스 배포
- 고객 피해 방지 기준: feature.md 3번 4항목 + 검수 미신청 건 "수량 미확인" 오해 방지 표시
- 실제 한 건의 QA 예시: 미니가전 셀러 A — 1688 링크 1건 ×100, 유료 검수(수량+외관) 신청 → 입고 완료, 사진 2장, 외포장 정상, 수량 100/100·하자 0 → 견적: 상품가+수수료(5%+VAT)+검수비+예상 운임(12kg/0.08CBM) 항목별 표시
- 통과해도 실패인 경우: ①입고 사진/검수 결과가 실제와 다름 ②견적에 없던 비용이 나중에 붙음 ③검수 미신청 셀러가 수량 확인된 것으로 오인하는 화면

## 4. PM 후보 판단

- PM이 판단하고 진행할 수 있는 것: 두리무역 벤치마크 범위 내 화면 구성·문구, 합격 예시·QA 시나리오
- 대표님 판단이 필요한 것: 요율 컷팅(추후), 검수 정책 변경, 원격 배포·push
- 후보가 너무 클 수 있는 이유: 셀러 가입/로그인 + 주문 접수 + 운영자 입고·검수 입력 + 견적 계산 + 대시보드 = 화면 4~5개 + 데이터 모델 전체가 한 덩어리. 두리무역 벤치마크(대시보드 12메뉴)를 다 따라가면 폭발
- 후보가 너무 작을 수 있는 이유: 없음 (최소로 잡아도 신뢰 고리 전체가 필요)
- 같은 고객 약속 안에서 묶을 수 있는 인접 고객 결과: 창고 주소 발급(입고ID) — 주문 접수의 전제라 포함 고려
- 절대 묶으면 안 되는 다른 고객 약속: LCL 국제배송·통관·추적(Feature 2), 1688 검색·구매 자동화
- Team Leader에게 요청하는 범위 판단: 이 후보를 한 Phase로 갈지, 세로 분할(예: 접수+입고확인 먼저 / 견적 다음)할지 `superpowers:brainstorming`으로 판단해 달라. 단 분할해도 각 Phase는 고객이 보는 결과로 끝나야 한다

## 5. Team Leader 범위 판단 후 PM 반영

- Team Leader 판단: **너무 큼** (`01-teamleader-intake.md` §3)
- PM 재조정 내용: 같은 고객 약속(Feature 1) 안에서 3개 Phase로 세로 분할 —
  - **Phase 1 (이 packet)**: 셀러가 두리무역 외관의 사이트에서 가입·로그인하고, 주문(링크·수량·옵션·구분·유료 검수 선택)을 접수하고, 내 주문 목록·상태를 본다. 데이터 격리 포함.
  - **Phase 2**: 운영자 입고·확인 기록 → 셀러 대시보드에서 입고 증거 확인 (창고 주소/입고ID 발급 포함)
  - **Phase 3**: 항목별 견적 확인 (두리무역 요율 초기값) → feature.md 6번 합격 예시 전체 통과 = Feature 1 완료
- Phase 1 완료 기준(좁힘): 합격 예시 중 "주문 접수" 구간 — 셀러 A가 가입·로그인 → 1688 링크 ×100 + 유료 검수 선택으로 접수 → 내 주문 목록에서 접수 상태 확인, 타 셀러에게 안 보임. 입고·견적 확인은 Phase 2·3 완료 기준.
- Phase 경계 확정 여부: **확정** (PM, 2026-07-04)
- 대표님 재정렬 필요 여부: 없음 (고객 약속·제외 범위·만족 기준 무변경, 순서만 분할)
