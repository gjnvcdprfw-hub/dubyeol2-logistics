# Phase 9 고객 결과 요청

상태: PM 후보 작성 완료 (2026-07-06) → Team Leader 범위 판단 완료

이 파일은 PM이 Team Leader에게 넘기는 후보 고객 결과 요청이다. 구현 티켓이 아니라 고객 결과 단위 작업만 적는다.

PM은 이 파일만으로 Phase 경계를 최종 확정하지 않는다. Team Leader가 `superpowers:brainstorming`으로 범위를 판단한 뒤, PM이 Phase 경계를 확정한다.

## 1. 연결 맥락

- 연결된 고객 흐름: 흐름 1 — 주문을 맡기고 입고·검수·견적을 확인 / 흐름 2 — 출고와 배송으로 이어지는 기반 / 흐름 3 — 건별 비용·상태 확인
- 연결된 메뉴탭 / 화면 영역: 셀러 주문 접수, 셀러 주문 상세, 운영자 입고/검수, 운영자 견적, 셀러 견적 확인
- 연결된 기능 약속: Feature 5 — SKU 작업·정산 기초 + 예치금 충전 + 출고 포장/패킹리스트 기초
- 연결된 Phase: Phase 9

## 1.1 플랫폼 독립 원칙 확인

- 고객 계약: 고객이 맡긴 물건이 상품별·SKU별로 구분되고, 입고·검수·견적·정산 근거가 같은 SKU 기준으로 남는다.
- 이번 Phase의 입력 방식 후보: 웹 주문 폼, 운영자 입고/검수 폼, 운영자 견적 폼
- 이번 Phase의 출력 방식 후보: 웹 대시보드, 주문 상세, 운영자 주문 상세, 견적 카드
- 이번 Phase의 실행 엔진 후보: 무관
- 고객 계약에서 제외할 구현 선택지: 포장단위 마커, 패킹리스트 PDF, 실송장 발급, 외부 배송조회 API, 1688 API/OAuth/크롤링, 자동 구매
- 입력 방식이 바뀌어도 유지되어야 하는 고객 결과: SKU별 수량·입고·검수·견적·정산 근거가 섞이지 않는다.
- 출력 방식이 바뀌어도 유지되어야 하는 고객 결과: 고객과 운영자가 같은 SKU 기준을 본다.
- 실행 엔진이 바뀌어도 유지되어야 하는 고객 결과: 작업·정산의 최소 단위가 주문 전체 한 줄이 아니라 SKU 라인이다.

## 2. Phase 고객 목표

- 이번 Phase에서 고객이 끝낼 수 있어야 하는 일: 한 주문 묶음 안에 여러 상품/SKU를 넣고, SKU별 입고·검수·견적·정산 근거를 확인한다.
- 이 일이 안 되면 고객에게 생기는 피해: 옵션별 단가, 수량, 하자, 누락, 검수비가 섞여 고객이 무엇에 얼마가 청구됐는지 믿을 수 없다.
- 이번 Phase가 끝나면 고객이 다음으로 할 수 있는 일: SKU별 정산 근거 위에 예치금 충전/차감과 출고 포장단위/패킹리스트를 연결할 수 있다.

## 3. 고객 결과 요청

### 고객 결과 요청 1

- 고객에게 생기는 변화: 로그인한 셀러와 운영자가 한 주문 묶음 안의 상품별·SKU별 수량, 입고, 검수, 견적, 정산 근거를 구분해 본다.
- 완료 기준:
  - 주문 묶음 1건에 상품 라인 여러 개를 담을 수 있다.
  - 상품 라인 안에 SKU 라인 여러 개를 담을 수 있다.
  - SKU 라인에는 옵션명, 수량, 단가 기준, 입고수량, 부족수량, 하자수량, 검수 결과가 분리되어 남는다.
  - 운영자는 SKU별 입고/검수 결과를 기록할 수 있다.
  - 운영자는 SKU별 견적 근거를 기록하거나 확인할 수 있다.
  - 셀러는 주문 상세에서 SKU별 입고·검수·견적 근거와 주문 묶음 합계를 함께 본다.
  - 다른 셀러의 SKU 라인, 입고수량, 검수결과, 견적 근거는 보이지 않는다.
- 이번에 안 하는 것: 포장단위 마커, 박스별 무게/CBM, 패킹리스트 생성, 실송장 발급, 예치금 충전, 출고 상태 추적, 장바구니·내상품·스마트오더, 1688 외부 연동.
- 고객 피해 방지 기준:
  - SKU별 수량·하자·부족이 주문 전체 한 줄로 뭉치면 실패.
  - SKU별 단가가 다른데 정산 근거가 주문 전체 한 줄로만 보이면 실패.
  - 유료 검수 결과가 SKU별로 남지 않으면 실패.
  - Phase 9에서 포장단위가 확정된 것처럼 보이면 실패.
  - 다른 셀러의 SKU 데이터가 보이면 실패.
- 실제 한 건의 QA 예시:
  - 셀러 A가 주문 묶음 1건을 만든다.
  - 상품 A에는 SKU 2개가 있다: 빨강 50개, 파랑 50개.
  - 상품 B에는 SKU 1개가 있다: L 사이즈 30개.
  - 운영자가 입고/검수 결과를 기록한다: 빨강 50개 정상, 파랑 45개 입고/5개 부족, L 사이즈 30개 중 2개 하자.
  - 셀러 주문 상세에는 SKU 3개가 각각 분리되어 보이고, 부족/하자/정상 수량이 섞이지 않는다.
  - 견적에는 SKU별 상품가·수량·검수비·중국배송비 배분 근거와 주문 묶음 합계가 함께 보인다.
  - 화면에는 포장단위가 아직 확정되지 않았고, 출고요청 때 배정된다는 상태가 보인다.
- 통과해도 실패인 경우: SKU별 입고·검수는 보이지만 정산이 주문 전체 한 줄로만 남거나, 포장단위가 Phase 9에서 확정된 것처럼 보이는 경우.

## 4. PM 후보 판단

- PM이 판단하고 진행할 수 있는 것: 주문 묶음/상품 라인/SKU 라인 경계, SKU별 작업·정산 최소 필드, 기존 단건 주문의 호환 표시, 포장단위 보류 문구.
- 대표님 판단이 필요한 것: SKU별 정산 배분 정책이 사업적으로 달라져야 하는 경우, 실제 포장 정책, 실제 패킹리스트 양식, 실운영 요율 컷팅.
- 후보가 너무 클 수 있는 이유: SKU 구조와 동시에 포장단위, 패킹리스트, 예치금, 출고 추적까지 넣으면 Phase가 너무 커지고 실제 창고 작업 순서와 섞인다.
- 후보가 너무 작을 수 있는 이유: SKU 라인만 만들고 입고·검수·견적 화면이 SKU 기준으로 바뀌지 않으면 작업·정산 문제를 해결하지 못한다.
- 같은 고객 약속 안에서 묶을 수 있는 인접 고객 결과: 기존 단건 주문을 SKU 1개짜리 주문처럼 보이게 하는 호환, SKU별 검수/견적 표시, 주문 묶음 합계.
- 절대 묶으면 안 되는 다른 고객 약속: 포장단위 확정, 패킹리스트 생성, 실송장 발급, 예치금 충전, 외부 배송조회, 1688 자동 연동.
- Team Leader에게 요청하는 범위 판단: 이 후보가 Phase 9 한 Phase로 적정한지 `superpowers:brainstorming`으로 판단한다.
