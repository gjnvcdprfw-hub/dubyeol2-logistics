# Phase 9 Team Leader Intake

상태: 작성 완료 (2026-07-06). Superpowers: `superpowers:using-superpowers`, `superpowers:brainstorming`

이 파일은 Team Leader가 `superpowers:brainstorming`으로 후보 고객 결과 요청을 이해하고, 한 Phase로 진행하기에 너무 크거나 작은지 판단한 뒤 작성한다.

## 1. 입력

- 받은 고객 결과 요청: `00-customer-outcome.md`
- 승인된 고객 약속: 상품별·SKU별 작업과 정산이 섞이지 않게 하고, 이후 예치금·출고·포장·패킹리스트를 같은 기준 위에 올릴 수 있게 한다.
- 대표님 승인: "8은 일단 보류, 9-11까지 먼저 진행해" / "지금은 상품마다, sku마다 구분되어있어? 구분되어있어야 작업이랑 정산을 하디 않을까?" / "포장단위랑 패킹리스트는 출고요청때 붙이면 어때? 처음부터는 필요할까?" / "그래"
- 현재 Phase 9 경계: 주문 묶음 -> 상품 라인 -> SKU 라인 구조와 SKU별 입고·검수·견적·정산 근거.
- 보류 담장: 포장단위 마커, 패킹리스트 생성, 실송장 발급, 예치금 충전, 출고 상태 추적, 1688 외부 연동.
- 사용한 Superpowers: `superpowers:using-superpowers`, `superpowers:brainstorming`

## 2. 이해한 고객 결과

- 고객이 끝내려는 일: 고객과 운영자가 같은 주문 안에서도 상품별·SKU별 수량, 하자, 부족, 단가, 검수비, 견적 근거를 따로 확인한다.
- 고객 결과 요청을 잘못 이해하면 고객에게 생기는 피해: 옵션별 단가와 수량이 섞여 잘못 청구되거나, 하자/부족 책임을 설명하지 못한다.
- 이번 Phase에서 반드시 지켜야 하는 결과: SKU별 작업·정산 근거가 주문 전체 한 줄로 뭉치지 않는다.

## 2.1 플랫폼 독립 원칙 확인

- 고객 계약이 입력 방식에 의존하지 않는가: 예. 웹 폼, 엑셀, 외부 연동 여부와 무관하게 SKU별 작업·정산 단위가 유지되어야 한다.
- 고객 계약이 출력 방식에 의존하지 않는가: 예. 화면, 알림, 문서로 바뀌어도 고객과 운영자는 같은 SKU 기준을 봐야 한다.
- 고객 계약이 실행 엔진에 의존하지 않는가: 예. 구현 엔진과 무관하게 데이터 단위가 SKU 라인으로 분리되어야 한다.
- 입력/출력/실행 엔진이 구현 선택지로 분리되어 있는가: 예.
- 고객 계약 변경이 필요해 PM에게 되돌릴 사항: 없음.

## 3. Phase 범위 판단

- 판단: **적정**.
- 너무 크다면 PM에게 요청할 Phase 분리: 현재는 필요 없음. 단, 포장단위·패킹리스트·예치금·출고 추적을 같이 넣으면 즉시 너무 커진다.
- 너무 작다면 같은 고객 약속 안에서 함께 묶을 인접 고객 결과: DB 구조만 바꾸면 너무 작다. SKU별 입고·검수·견적·정산 표시와 기존 단건 주문 호환까지 묶어야 고객 결과가 난다.
- 아무 일이나 추가하지 않았는가: 예. SKU 작업·정산 근거에 직접 필요한 범위만 포함했다.
- 고객 약속, 제외 범위, 업무 범위, 만족 기준 변경 여부: 없음.
- PM 재조정 필요 여부: 없음.

## 4. 대안 비교

### A. 추천 — 주문 묶음 + 상품 라인 + SKU 라인

- 내용: 현재 `Order` 한 줄 구조를 주문 묶음으로 보고, 상품 라인과 SKU 라인을 분리한다. 입고·검수·견적·정산 근거는 SKU 라인 기준으로 남긴다.
- 장점: 실제 작업과 정산 단위가 맞고, Phase 10 예치금과 Phase 11 포장/패킹리스트를 올릴 기반이 된다.
- 단점: 기존 단건 주문 화면과 테스트를 호환 처리해야 한다.
- 판단: Phase 9 계약에 가장 맞다.

### B. 기존 `Order.optionText` 확장만 사용

- 내용: 옵션 텍스트에 색상/사이즈/수량을 계속 적고 화면만 조금 개선한다.
- 장점: 빠르다.
- 단점: SKU별 하자·부족·단가·정산을 구조적으로 계산할 수 없어 대표님이 지적한 문제를 해결하지 못한다.
- 판단: 불가.

### C. SKU + 포장단위 + 패킹리스트까지 한 번에 구현

- 내용: SKU 구조와 동시에 BOX-1/BOX-2, 박스별 무게/CBM, 패킹리스트 생성까지 만든다.
- 장점: 출고 업무까지 빨리 이어진다.
- 단점: 포장은 실제 출고 때 바뀌는 단위라 지금 확정하면 창고 작업과 충돌한다. Phase 9가 너무 커진다.
- 판단: Phase 11로 분리.

## 5. 위험 점검

- 숨은 위험: 기존 단건 주문을 새 SKU 구조로 어떻게 보이게 할지 놓치면 기존 Phase 1~7 흐름이 깨질 수 있다.
- 이번 Phase 범위를 넘는 유혹: 예치금 충전, 포장단위, 패킹리스트, 장바구니, 1688 연동을 같이 넣는 것.
- 통과해도 실패인 경우: SKU 테이블은 생겼지만 화면과 정산이 여전히 주문 전체 한 줄 기준이면 실패다.
- PM에게 되돌릴 질문: 없음.

## 6. Superpowers 준수 확인

- `superpowers:brainstorming`을 사용했는가: 예.
- `superpowers:writing-plans`로 넘어갈 준비가 됐는가: 예.
- 구현은 Builder 서브에이전트에게 맡겨야 함을 확인했는가: 예.
- 고객 약속, 이번에 안 하는 것, 업무 범위, 만족 기준이 바뀌었는가: 아니오.
- PM에게 되돌릴 질문: 없음.

## 7. 다음 단계 판정

- `superpowers:writing-plans` 진입 가능 여부: **가능**.
- 근거: Phase 9 경계가 SKU 작업·정산 기초로 닫혔고, 포장단위·패킹리스트·예치금·출고 추적은 명확히 제외됐다.
