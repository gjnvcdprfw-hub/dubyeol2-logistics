## 1. 입력

- PM 고객 결과 요청: `00-customer-outcome.md`
- 승인된 고객 약속: 로그인한 셀러가 견적 완료 주문을 출고 요청하고, 예치금 차감·내역·주문 상태·출고관리 상태를 확인한다.
- 대표님 승인: "로그인후 셀러기능 구현하자," → PM 범위/합격 기준 제안 → "그래"
- 현재 Phase 7 경계: 견적 후 출고 요청 + 예치금 차감 + 출고 상태 확인
- 보류 담장: 실계좌, 실입금, 실결제, 송장 발급, 통관 신고, 실제 창고 출고, 반품/환불, 스마트오더/장바구니/내상품 고도화

## 2. 프로젝트 맥락 확인

- 기존 데이터 모델: `User`, `Order`, `InboundPhoto`. `Order.status`는 문자열이고 현재 `REQUESTED`, `RECEIVED` 중심으로 쓰인다.
- 기존 견적 데이터: `Order`에 `quote*` 필드와 `quotedAt`가 있고, `src/lib/quote.ts`의 `computeQuote()`로 원화 합계를 계산한다.
- 기존 셀러 격리: 주문 상세와 목록은 `sellerId: session.userId` 스코프를 사용한다.
- 기존 예치금 화면: `src/app/dashboard/wallet/page.tsx`는 준비 중 셸만 있고 원장 모델은 없다.
- 기존 출고 화면: `src/app/dashboard/shipment/page.tsx`는 상태 타일 5개와 빈 상태만 있다.
- 필요한 변화: 주문 상태 확장, 예치금 잔액/거래 원장, 출고 요청 처리, 셀러 화면 4곳 정합화.

## 3. 범위 판단

- 판단: **적정**.
- 이유: 데이터 모델, 서버 처리, 주문 상세, 예치금, 출고관리, 대시보드 홈을 함께 건드리지만 전부 한 고객 결과("견적 후 보내달라 결정을 끝낸다")에 묶인다.
- 너무 큰 범위가 되지 않는 이유: 실계좌·실입금·실결제·송장·통관·택배 추적을 명시적으로 제외했다.
- 너무 작은 범위가 아닌 이유: 출고 요청만 만들지 않고 예치금 차감과 내역, 주문 상태, 출고관리 확인까지 묶어 고객이 비용 정산을 믿을 수 있게 한다.
- PM 복귀 필요 여부: **없음**. 고객 약속, 제외 범위, 합격 기준, 담장이 바뀌지 않았다.

## 4. 대안 비교

### A. 추천 — 기존 `Order` 중심 + 새 예치금 원장

- 내용: `Order.status`에 출고 요청 상태를 추가하고, `WalletLedger` 같은 예치금 거래 원장을 새로 둔다. 출고 요청 API는 주문 소유자·견적 완료·잔액 충분 여부를 확인한 뒤 한 트랜잭션에서 예치금 차감 내역과 주문 상태를 같이 저장한다.
- 장점: 기존 주문/견적 흐름을 가장 적게 바꾸면서 돈/상태 불일치를 막는다.
- 단점: 예치금 원장 모델과 시드/테스트가 필요하다.
- 판단: Phase 7 계약에 가장 맞다.

### B. 주문 상태만 먼저 구현

- 내용: 출고 요청 버튼과 주문 상태 변경만 만들고 예치금은 준비 중으로 둔다.
- 장점: 빠르다.
- 단점: 대표님이 승인한 합격 기준(예치금 차감·내역)이 빠져 실패다.
- 판단: 불가.

### C. 실결제/충전까지 함께 구현

- 내용: 계좌/입금/결제사/환불까지 확장한다.
- 장점: 운영 완성에 가까워진다.
- 단점: 돈·계약·외부 연동·실서비스 위험이 섞여 두별2 금지선과 이번 담장을 넘는다.
- 판단: 후속 Phase.

## 5. Phase 7에서 하는 일

1. 예치금 잔액과 거래 내역을 로컬 테스트 원장으로 관리한다.
2. 견적 완료 주문의 총 원화 금액을 기존 `computeQuote()` 기준으로 계산한다.
3. 셀러가 주문 상세에서 출고 요청을 누르면 잔액 충분 여부를 확인한다.
4. 잔액이 충분하면 한 트랜잭션에서 예치금 차감 내역과 주문 상태를 함께 저장한다.
5. 잔액이 부족하면 주문 상태를 바꾸지 않고 부족 안내만 보여준다.
6. 출고관리, 예치금 관리, 대시보드 홈, 주문 상세이 같은 사실을 보여주게 한다.
7. 다른 셀러의 주문, 잔액, 예치금 내역이 보이지 않도록 테스트로 확인한다.

## 6. 위험과 대응

- 위험 1: 돈만 차감되고 주문 상태가 안 바뀜 → 트랜잭션과 회귀 테스트로 묶는다.
- 위험 2: 상태만 바뀌고 예치금 내역이 없음 → 출고 요청 API의 완료 조건을 원장 기록까지 포함한다.
- 위험 3: 잔액 부족인데 출고 요청이 완료됨 → 잔액 부족 테스트와 UI 안내를 둔다.
- 위험 4: 다른 셀러 데이터 노출 → 모든 조회·변경에 `sellerId` 스코프를 유지하고 격리 테스트를 둔다.
- 위험 5: 실결제처럼 오해 → 예치금 화면과 버튼 문구에 로컬 테스트/오픈 전 확정 담장을 유지한다.
