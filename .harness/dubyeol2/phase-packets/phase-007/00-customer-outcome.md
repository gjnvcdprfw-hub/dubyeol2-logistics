## 1. 연결 맥락

- 연결된 고객 흐름: 흐름 2 — 검수 완료 물량을 LCL 국제배송·통관으로 한국까지
- 연결된 메뉴탭 / 화면 영역: 셀러 주문 상세, 셀러 출고관리, 셀러 예치금 관리, 셀러 대시보드 홈
- 연결된 기능 약속: Feature 4 — 로그인 후 셀러 출고 요청·예치금 흐름
- 연결된 Phase: Phase 7

## 1.1 플랫폼 독립 원칙 확인

- 고객 계약: 셀러가 견적 완료 주문을 보고 출고를 요청하면, 예치금 차감과 출고 상태가 같은 사실로 기록되어 고객이 "보내달라" 결정을 끝낸다.
- 이번 Phase의 입력 방식 후보: 웹 버튼, 주문 상세 화면, 예치금 관리 화면
- 이번 Phase의 출력 방식 후보: 웹 대시보드, 주문 상태, 예치금 내역
- 이번 Phase의 실행 엔진 후보: 무관
- 고객 계약에서 제외할 구현 선택지: 특정 결제사, 실제 은행 계좌, 실시간 배송조회 API, WMS, 통관 API, 택배 송장 API
- 입력 방식이 바뀌어도 유지되어야 하는 고객 결과: 출고 요청과 예치금 차감이 같은 주문에 일관되게 남는다.
- 출력 방식이 바뀌어도 유지되어야 하는 고객 결과: 고객은 출고 요청 상태와 예치금 차감 내역을 확인할 수 있다.
- 실행 엔진이 바뀌어도 유지되어야 하는 고객 결과: 돈 흐름과 주문 상태가 엇갈리지 않는다.

## 2. Phase 고객 목표

- 이번 Phase에서 고객이 끝낼 수 있어야 하는 일: 견적 완료 주문 1건을 출고 요청하고, 예치금 차감과 상태 변경을 확인한다.
- 이 일이 안 되면 고객에게 생기는 피해: 셀러는 견적을 보고도 물건을 보낼 수 없고, 서비스는 실제 매출 흐름으로 이어지지 않는다.
- 이번 Phase가 끝나면 고객이 다음으로 할 수 있는 일: 운영자가 출고 준비, 송장, 통관, 국내 배송 추적을 처리하는 후속 Phase로 넘어갈 수 있다.

## 3. 고객 결과 요청

### 고객 결과 요청 1

- 고객에게 생기는 변화: 로그인한 셀러가 견적 완료 주문 상세에서 출고 요청을 누르면 예치금이 차감되고, 예치금 내역과 주문 상태와 출고관리 화면이 같은 상태를 보여준다.
- 완료 기준:
  - 견적 완료 주문 1건에서 출고 요청을 완료할 수 있다.
  - 예치금 잔액이 견적 금액만큼 차감된다.
  - 예치금 내역에 주문번호, 금액, 차감 사유, 시간, 상태가 남는다.
  - 주문 상태가 출고 요청/출고 준비 상태로 바뀐다.
  - 출고관리 화면과 대시보드 홈 카드가 같은 상태를 반영한다.
  - 다른 셀러의 주문, 잔액, 예치금 내역은 보이지 않는다.
- 이번에 안 하는 것: 실입금, 실계좌 노출, PG/간편결제, 외부 배송조회, 송장 발급, 통관 신고, 실제 창고 출고 처리, 반품/환불, 스마트오더/장바구니/내상품 고도화.
- 고객 피해 방지 기준:
  - 돈만 차감되고 상태가 안 바뀌면 실패.
  - 상태만 바뀌고 예치금 내역이 없으면 실패.
  - 잔액 부족인데 출고 요청이 완료되면 실패.
  - 다른 셀러 데이터가 보이면 실패.
  - 실결제처럼 오해하게 보이면 실패.
- 실제 한 건의 QA 예시:
  - 셀러 A에게 예치금 300,000원을 준비한다.
  - 셀러 A의 견적 완료 주문 1건 총액이 118,400원이다.
  - 셀러 A가 주문 상세에서 출고 요청을 누른다.
  - 예치금 잔액은 181,600원으로 바뀐다.
  - 예치금 내역에 `출고 요청 차감 118,400원`이 남는다.
  - 주문 상세와 출고관리에는 같은 주문이 `출고 요청` 또는 `출고 준비` 상태로 보인다.
- 통과해도 실패인 경우: 돈만 차감되고 주문 상태가 안 바뀌거나, 상태만 바뀌고 예치금 내역이 없거나, 다른 셀러 주문/잔액이 보이는 경우.

## 4. PM 후보 판단

- PM이 판단하고 진행할 수 있는 것: 로컬 테스트 예치금 원장, 상태명, 화면 흐름, 잔액 부족 처리, 기존 주문/견적 흐름과의 연결.
- 대표님 판단이 필요한 것: 실제 계좌, 실제 충전 방식, 결제사, 출고비 정산 정책, 환불 정책, 실운영 문구.
- 후보가 너무 클 수 있는 이유: 예치금 충전, 결제사 연동, 송장/통관/택배 추적까지 넣으면 돈·외부계약·운영 정책이 섞인다.
- 후보가 너무 작을 수 있는 이유: 출고 요청만 만들고 예치금 내역을 만들지 않으면 고객은 비용 정산을 믿을 수 없다.
- 같은 고객 약속 안에서 묶을 수 있는 인접 고객 결과: 잔액 부족 안내, 예치금 내역, 대시보드 출고 상태 카드.
- 절대 묶으면 안 되는 다른 고객 약속: 실제 입금확인 자동화, 실제 계좌 노출, PG 연동, 송장/통관/택배 추적, 반품/환불.
- Team Leader에게 요청하는 범위 판단: 이 후보가 Phase 7 한 Phase로 적정한지 `superpowers:brainstorming`으로 판단한다.
