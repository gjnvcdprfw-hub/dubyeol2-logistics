# Phase 11 고객 결과 요청

상태: PM 후보 작성 완료 (2026-07-06)

이 파일은 PM이 Team Leader에게 넘기는 후보 고객 결과 요청이다. 구현 티켓이 아니라 고객 결과 단위 작업만 적는다.

PM은 이 파일만으로 Phase 경계를 최종 확정하지 않는다. Team Leader가 `superpowers:brainstorming`으로 범위를 판단한 뒤, PM이 Phase 경계를 확정한다.

## 1. 연결 맥락

- 연결된 고객 흐름: 흐름 2 — 검수 완료 물량을 LCL 국제배송·통관으로 한국까지.
- 연결된 메뉴탭 / 화면 영역: 출고관리, 주문 상세, 대시보드 홈, 운영자 출고 처리 화면.
- 연결된 기능 약속: Feature 5 — SKU 작업·정산 기초 + 예치금 충전 + 출고 포장/패킹리스트 기초.
- 연결된 Phase: Phase 11.
- 직전 Phase 결과: Phase 9에서 주문 묶음 -> 상품 라인 -> SKU 라인 구조가 완료됐고, Phase 10에서 예치금 충전 요청·운영자 확인·잔액/원장이 완료됐다.

## 1.1 플랫폼 독립 원칙 확인

- 고객 계약: 고객이 출고 요청 이후 박스별 SKU 구성과 출고 상태를 확인하고, 운영자는 나중에 패킹리스트로 뽑을 수 있는 포장 근거를 남긴다.
- 이번 Phase의 입력 방식 후보: 운영자 박스 생성/수정 폼, 박스별 SKU 배정 폼, 수동 출고 상태 입력.
- 이번 Phase의 출력 방식 후보: 셀러 출고관리, 셀러 주문 상세, 운영자 출고 처리 화면, 패킹리스트 기초 데이터 표시.
- 이번 Phase의 실행 엔진 후보: 무관.
- 고객 계약에서 제외할 구현 선택지: 택배사 API, 세관 API, 외부 배송조회 API, 자동 송장 동기화, 실제 배송 완료 확정, 정식 PDF 자동 발행, 실서비스 변경, 유료 API 실호출.
- 입력 방식이 바뀌어도 유지되어야 하는 고객 결과: 어떤 SKU가 몇 개씩 어느 박스에 들어갔는지와 박스별 무게·CBM·출고 상태가 일치한다.
- 출력 방식이 바뀌어도 유지되어야 하는 고객 결과: 셀러와 운영자가 같은 포장단위와 출고 상태를 본다.
- 실행 엔진이 바뀌어도 유지되어야 하는 고객 결과: SKU 수량보다 더 많이 포장하거나, 다른 셀러의 박스 구성이 보이는 일이 없다.

## 2. Phase 고객 목표

- 이번 Phase에서 고객이 끝낼 수 있어야 하는 일: 출고요청이 된 주문의 SKU를 운영자가 박스 단위로 나누고, 셀러가 박스별 구성과 출고 상태를 확인한다.
- 이 일이 안 되면 고객에게 생기는 피해: 실제로 어떤 박스에 어떤 SKU가 들어갔는지 몰라 운임·분실·오배송·패킹리스트 대응 근거가 깨진다.
- 이번 Phase가 끝나면 고객이 다음으로 할 수 있는 일: 박스별 구성 데이터를 바탕으로 정식 패킹리스트 출력, 송장/통관/배송 추적 연동을 별도 Phase에서 붙일 수 있다.

## 3. 고객 결과 요청

### 고객 결과 요청 1

- 고객에게 생기는 변화: 운영자는 출고요청 주문의 SKU를 BOX-1, BOX-2 같은 포장단위에 배정하고 박스별 무게·CBM·상태를 남기며, 셀러는 출고관리와 주문 상세에서 그 결과를 본다.
- 완료 기준:
  - 출고요청 상태의 주문에 운영자가 포장단위를 만들 수 있다.
  - 각 포장단위에는 마커(예: `BOX-1`), 박스 메모, 무게 kg, CBM, 출고 상태, 생성/수정 시각이 남는다.
  - 운영자는 박스별로 해당 주문의 SKU와 수량을 배정할 수 있다.
  - 같은 SKU의 박스 배정 총량은 해당 SKU의 출고 가능한 수량을 넘을 수 없다.
  - 운영자는 포장단위 상태를 수동으로 바꿀 수 있다.
  - 셀러는 자기 주문의 박스별 SKU 구성·수량·무게·CBM·상태를 볼 수 있다.
  - 셀러는 다른 셀러의 포장단위와 SKU 구성을 볼 수 없다.
  - 화면은 정식 PDF 패킹리스트 자동 발행이나 실제 배송조회가 아직 아님을 혼동시키지 않는다.
- 이번에 안 하는 것: 외부 배송조회 API, 세관 API, 택배사 API, 자동 송장 동기화, 실제 배송 완료 확정, 정식 패킹리스트 PDF 자동 발행, 실제 송장 발급, 실서비스 변경, 정식 운영 전환.
- 고객 피해 방지 기준:
  - SKU 출고 가능 수량보다 더 많이 박스에 배정되면 실패.
  - 박스별 구성은 있는데 셀러 화면에 보이지 않으면 실패.
  - 셀러가 다른 셀러의 박스 구성이나 출고 상태를 보면 실패.
  - 운영자 상태와 셀러 상태가 서로 다르면 실패.
  - 정식 PDF 패킹리스트나 실제 배송조회가 되는 것처럼 보이면 실패.
- 실제 한 건의 QA 예시:
  - 셀러 A 주문은 출고요청 상태이고 SKU 3개가 있다.
  - 운영자가 `BOX-1`에 빨강 2개, 파랑 1개를 넣고 무게 12.5kg, CBM 0.08을 입력한다.
  - 운영자가 `BOX-2`에 파랑 1개, L 3개를 넣고 무게 18.2kg, CBM 0.13을 입력한다.
  - 운영자가 상태를 `포장완료`로 바꾸면 셀러 출고관리와 주문 상세에서 BOX-1/BOX-2 구성, 수량, 무게, CBM, 상태가 보인다.
  - 같은 SKU를 출고 가능 수량보다 더 많이 배정하려 하면 저장되지 않는다.
  - 셀러 B는 셀러 A의 박스 구성을 볼 수 없다.
- 통과해도 실패인 경우: 박스 마커만 있고 SKU별 수량이 없거나, 박스별 SKU 구성은 DB에 있는데 셀러가 확인할 수 없는 경우.

## 4. PM 후보 판단

- PM이 판단하고 진행할 수 있는 것: 포장단위 마커 규칙, 박스별 SKU 구성·수량·무게·CBM, 수동 출고 상태, 셀러/운영자 표시 범위, 패킹리스트 기초 데이터.
- 대표님 판단이 필요한 것: 실제 택배사/포워더 송장 정책, 정식 패킹리스트 양식, 외부 배송조회 연동, 세관/택배사 API, 실제 배송 완료 확정 기준.
- 후보가 너무 클 수 있는 이유: PDF 발행, 외부 배송조회, 송장/세관/택배사 API까지 붙이면 실제 운영 정책과 외부 연동이 필요해 Phase 11을 넘는다.
- 후보가 너무 작을 수 있는 이유: 박스 마커만 만들고 SKU 구성·수량·무게·CBM이 없으면 나중에 패킹리스트 근거가 되지 않는다.
- 같은 고객 약속 안에서 묶을 수 있는 인접 고객 결과: 셀러 출고관리 표시, 주문 상세 표시, 운영자 수동 출고 상태, 패킹리스트 기초 데이터.
- 절대 묶으면 안 되는 다른 고객 약속: 정식 패킹리스트 PDF 자동 발행, 실송장 발급, 외부 배송조회, 세관 API, 택배사 API, 실제 배송 완료 확정, Phase 8 정식 운영 전환.
- Team Leader에게 요청하는 범위 판단: 이 후보가 Phase 11 한 Phase로 적정한지 `superpowers:brainstorming`으로 판단한다.
