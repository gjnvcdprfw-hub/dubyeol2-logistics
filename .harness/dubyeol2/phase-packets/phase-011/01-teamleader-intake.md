# Phase 11 Team Leader Intake

상태: 작성 완료 (2026-07-06). Superpowers: `superpowers:using-superpowers`, `superpowers:brainstorming`

이 파일은 Team Leader가 `superpowers:brainstorming`으로 후보 고객 결과 요청을 이해하고, 한 Phase로 진행하기에 너무 크거나 작은지 판단한 뒤 작성한다.

## 1. 입력

- 받은 고객 결과 요청: `00-customer-outcome.md`
- 승인된 고객 약속: 출고요청 이후 SKU가 실제 박스 단위로 어떻게 나가는지 셀러와 운영자가 같은 근거를 본다.
- 대표님 승인: "8은 일단 보류, 9-11까지 먼저 진행해" / "포장단위별로 마커가 붙어서 나중에 패킹리스트도 만들어야해" / "포장단위랑 패킹리스트는 출고요청때 붙이면 어때? 처음부터는 필요할까?" → PM 추천 → "그래".
- 현재 Phase 11 경계: 출고요청 주문의 포장단위 마커, 박스별 SKU 구성·수량, 무게, CBM, 운영자 수동 출고 상태, 셀러 화면 반영, 패킹리스트 기초 데이터.
- 보류 담장: 정식 PDF 패킹리스트 자동 발행, 실송장 발급, 외부 배송조회, 세관 API, 택배사 API, 자동 송장 동기화, 실제 배송 완료 확정, Phase 8 정식 운영 전환.
- 사용한 Superpowers: `superpowers:using-superpowers`, `superpowers:brainstorming`

## 2. 이해한 고객 결과

- 고객이 끝내려는 일: 출고 요청이 된 SKU들이 어떤 박스에 몇 개씩 들어갔는지 보고, 이후 패킹리스트와 운임/분실 대응의 근거를 남긴다.
- 고객 결과 요청을 잘못 이해하면 고객에게 생기는 피해: 박스별 구성 없이 출고 상태만 바뀌면 실제로 무엇이 나갔는지 확인할 수 없고, SKU 수량보다 많이 포장되면 정산·배송·분실 대응 근거가 깨진다.
- 이번 Phase에서 반드시 지켜야 하는 결과: 포장단위는 SKU 수량과 연결되어야 하고, 셀러와 운영자가 같은 박스별 SKU 구성·무게·CBM·상태를 봐야 한다.

## 2.1 플랫폼 독립 원칙 확인

- 고객 계약이 입력 방식에 의존하지 않는가: 예. 웹 폼, 엑셀, WMS 연동으로 바뀌어도 박스별 SKU 구성과 수량은 일치해야 한다.
- 고객 계약이 출력 방식에 의존하지 않는가: 예. 화면, PDF, 엑셀로 바뀌어도 같은 포장단위 근거를 보여야 한다.
- 고객 계약이 실행 엔진에 의존하지 않는가: 예. 구현 엔진과 무관하게 SKU 수량 초과 배정과 셀러 간 노출이 없어야 한다.
- 입력/출력/실행 엔진이 구현 선택지로 분리되어 있는가: 예.
- 고객 계약 변경이 필요해 PM에게 되돌릴 사항: 없음.

## 3. Phase 범위 판단

- 판단: **적정**.
- 너무 크다면 PM에게 요청할 Phase 분리: PDF 자동 발행, 실송장, 외부 배송조회, 세관/택배사 API, 실제 배송 완료 확정까지 같이 넣으면 실제 운영 정책과 외부 연동이 필요해 너무 크다.
- 너무 작다면 같은 고객 약속 안에서 함께 묶을 인접 고객 결과: 포장단위 마커만 만들고 SKU 구성·수량·무게·CBM·셀러 표시가 없으면 나중에 패킹리스트 근거가 되지 않는다.
- 아무 일이나 추가하지 않았는가: 예. 출고요청 이후 포장단위와 패킹리스트 기초에 직접 필요한 범위만 포함했다.
- 고객 약속, 제외 범위, 업무 범위, 만족 기준 변경 여부: 없음.
- PM 재조정 필요 여부: 없음.

## 4. 대안 비교

### A. 추천 — 운영자 수동 포장단위 + 박스별 SKU 구성 + 셀러 표시

- 내용: 출고요청 주문에 `ShipmentPackage`와 `ShipmentPackageItem`을 추가하고, 운영자가 박스 마커, 무게, CBM, 상태, SKU별 배정 수량을 저장한다. 셀러 주문 상세과 출고관리에는 같은 박스별 구성을 보여준다.
- 장점: 외부 연동 없이도 실제 창고 작업 근거가 남고, 이후 패킹리스트 PDF나 엑셀 출력으로 자연스럽게 이어진다.
- 단점: 운영자가 수동 입력해야 한다.
- 판단: Phase 11 계약에 가장 맞다.

### B. 박스 마커만 추가

- 내용: 주문에 `BOX-1`, `BOX-2` 같은 마커만 달고 상세 SKU 구성은 남기지 않는다.
- 장점: 빠르다.
- 단점: 어떤 SKU가 어느 박스에 몇 개 들어갔는지 모르므로 패킹리스트 근거가 되지 않는다. 고객 약속을 만족하지 못한다.
- 판단: 불가.

### C. 정식 패킹리스트 PDF·송장·배송조회까지 연결

- 내용: 박스별 구성으로 PDF를 자동 생성하고, 송장/배송조회/세관/택배사 API까지 연결한다.
- 장점: 운영 자동화에 가깝다.
- 단점: 실제 양식, 외부 API, 실서비스 정책, 돈/배송 책임 기준이 필요하다. Phase 8 보류와 금지선에 걸린다.
- 판단: Phase 11에서 제외.

## 5. 위험 점검

- 숨은 위험: SKU 배정 총량이 출고 가능한 수량을 초과하면 박스별 구성과 실제 주문 수량이 불일치한다.
- 숨은 위험: 출고요청 상태가 아닌 주문에도 박스가 생성되면 창고 작업 시점이 뒤섞인다.
- 숨은 위험: 관리자 화면에서 포장단위를 만들었지만 셀러 화면에 보이지 않으면 고객은 출고 근거를 확인할 수 없다.
- 이번 Phase 범위를 넘는 유혹: PDF 자동 생성, 송장, 운송장 회신 엑셀, 배송조회, 통관 상태 자동화.
- 통과해도 실패인 경우: 박스 마커는 보이지만 SKU 수량이 없거나, DB에는 박스 구성이 있는데 셀러 출고관리에는 보이지 않는 경우.
- PM에게 되돌릴 질문: 없음.

## 6. Superpowers 준수 확인

- `superpowers:using-superpowers`를 사용했는가: 예.
- `superpowers:brainstorming`을 사용했는가: 예.
- `superpowers:writing-plans`로 넘어갈 준비가 됐는가: 예.
- 구현은 Builder 서브에이전트에게 맡겨야 함을 확인했는가: 예.
- 고객 약속, 이번에 안 하는 것, 업무 범위, 만족 기준이 바뀌었는가: 아니오.
- PM에게 되돌릴 질문: 없음.
