# Phase 10 Team Leader Intake

상태: 작성 완료 (2026-07-06). Superpowers: `superpowers:using-superpowers`, `superpowers:brainstorming`

이 파일은 Team Leader가 `superpowers:brainstorming`으로 후보 고객 결과 요청을 이해하고, 한 Phase로 진행하기에 너무 크거나 작은지 판단한 뒤 작성한다.

## 1. 입력

- 받은 고객 결과 요청: `00-customer-outcome.md`
- 승인된 고객 약속: 셀러가 예치금 충전을 요청하고, 운영자가 수동 확인해 잔액과 원장을 맞춘다.
- 대표님 승인: "8은 일단 보류, 9-11까지 먼저 진행해" / Phase 10은 feature.md에서 "예치금 충전 신청·운영자 확인"으로 잠김.
- 현재 Phase 10 경계: 충전 요청, 충전 대기/승인/거절, 운영자 확인, 잔액 반영, 예치금 원장 기록.
- 보류 담장: 실계좌 실제값, PG/간편결제, 은행 API, 자동 입금 확인, 환불/세금계산서 정책, 정식 운영 전환, 포장단위/패킹리스트.
- 사용한 Superpowers: `superpowers:using-superpowers`, `superpowers:brainstorming`

## 2. 이해한 고객 결과

- 고객이 끝내려는 일: 출고 요청 전에 돈을 넣겠다는 요청을 남기고, 운영자 확인 뒤 잔액이 올라갔는지 원장과 화면에서 확인한다.
- 고객 결과 요청을 잘못 이해하면 고객에게 생기는 피해: 승인 전 잔액이 올라가거나, 승인됐는데 원장이 없거나, 같은 요청이 중복 승인되어 잔액이 부풀 수 있다.
- 이번 Phase에서 반드시 지켜야 하는 결과: 승인된 충전만 잔액에 한 번 반영되고, 승인/거절 상태와 원장이 셀러 화면과 운영자 화면에서 같은 사실로 남는다.

## 2.1 플랫폼 독립 원칙 확인

- 고객 계약이 입력 방식에 의존하지 않는가: 예. 웹 폼, 엑셀, 외부 연동으로 바뀌어도 충전 요청과 운영자 확인 결과가 잔액·원장과 맞아야 한다.
- 고객 계약이 출력 방식에 의존하지 않는가: 예. 화면, 알림, 문서로 바뀌어도 셀러와 운영자가 같은 충전 상태를 봐야 한다.
- 고객 계약이 실행 엔진에 의존하지 않는가: 예. 구현 엔진과 무관하게 승인 전/후 돈 상태가 일관되어야 한다.
- 입력/출력/실행 엔진이 구현 선택지로 분리되어 있는가: 예.
- 고객 계약 변경이 필요해 PM에게 되돌릴 사항: 없음.

## 3. Phase 범위 판단

- 판단: **적정**.
- 너무 크다면 PM에게 요청할 Phase 분리: 실제 계좌, PG/간편결제, 은행 API, 자동 입금 확인, 환불/세금계산서를 같이 넣으면 즉시 너무 커진다.
- 너무 작다면 같은 고객 약속 안에서 함께 묶을 인접 고객 결과: 충전 요청만 만들고 운영자 승인/거절, 잔액 반영, 원장 기록, 대시보드 반영이 없으면 고객 결과가 나지 않는다.
- 아무 일이나 추가하지 않았는가: 예. 예치금 충전 확인에 직접 필요한 범위만 포함했다.
- 고객 약속, 제외 범위, 업무 범위, 만족 기준 변경 여부: 없음.
- PM 재조정 필요 여부: 없음.

## 4. 대안 비교

### A. 추천 — 충전 요청 + 운영자 수동 승인/거절 + 기존 원장 반영

- 내용: `WalletTopUpRequest` 같은 요청 모델을 추가하고, 셀러 예치금 화면에서 충전 요청을 만들며, 운영자 화면에서 승인/거절한다. 승인 시 기존 `WalletTransaction`에 충전 credit을 만들고 잔액을 맞춘다.
- 장점: 실제 입금/결제 정책 없이도 고객이 충전 상태와 잔액을 확인할 수 있다. Phase 7의 예치금 차감 원장을 재사용한다.
- 단점: 실제 입금 자동 매칭은 아직 되지 않는다.
- 판단: Phase 10 계약에 가장 맞다.

### B. 기존 QA seed 함수만 화면에 노출

- 내용: `recordWalletCredit()`을 셀러 또는 운영자 버튼에서 바로 호출해 잔액만 올린다.
- 장점: 빠르다.
- 단점: 충전 요청 상태, 운영자 확인, 승인/거절 근거가 남지 않아 고객이 "내가 요청한 돈이 처리됐는지" 확인할 수 없다. 중복 처리 추적도 약하다.
- 판단: 불가.

### C. 실제 계좌/PG/은행 API까지 연결

- 내용: 실계좌 안내, 입금 자동확인, PG/간편결제, 환불/세금계산서까지 붙인다.
- 장점: 운영 자동화에 가까워진다.
- 단점: 외부 계약, 돈 정책, 실서비스 변경 승인, 유료/상태 변경 API가 필요하다. Phase 8 보류와 금지선에 걸린다.
- 판단: Phase 10에서 제외.

## 5. 위험 점검

- 숨은 위험: 기존 `recordWalletCredit()`은 테스트 seed 성격이라 그대로 public/admin route에 연결하면 감사 추적과 중복 승인 방지가 부족할 수 있다.
- 이번 Phase 범위를 넘는 유혹: 실제 계좌번호, 자동 입금 확인, 결제사, 환불/세금계산서, 정식 운영 문구를 같이 넣는 것.
- 통과해도 실패인 경우: 잔액은 늘었지만 충전 요청 상태가 그대로 대기거나, 충전 요청은 승인됐는데 원장이 없는 경우.
- PM에게 되돌릴 질문: 없음.

## 6. Superpowers 준수 확인

- `superpowers:brainstorming`을 사용했는가: 예.

- 구현은 Builder 서브에이전트에게 맡겨야 함을 확인했는가: 예.
- 고객 약속, 이번에 안 하는 것, 업무 범위, 만족 기준이 바뀌었는가: 아니오.
- PM에게 되돌릴 질문: 없음.
