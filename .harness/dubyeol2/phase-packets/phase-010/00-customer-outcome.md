# Phase 10 고객 결과 요청

상태: PM 후보 작성 완료 (2026-07-06)

이 파일은 PM이 Team Leader에게 넘기는 후보 고객 결과 요청이다. 구현 티켓이 아니라 고객 결과 단위 작업만 적는다.

PM은 이 파일만으로 Phase 경계를 최종 확정하지 않는다. Team Leader가 `superpowers:brainstorming`으로 범위를 판단한 뒤, PM이 Phase 경계를 확정한다.

## 1. 연결 맥락

- 연결된 고객 흐름: 흐름 3 — 건별 비용·상태 확인, 예치금 충전·정산
- 연결된 메뉴탭 / 화면 영역: 예치금 관리, 대시보드 홈, 운영자 예치금 확인 화면
- 연결된 기능 약속: Feature 5 — SKU 작업·정산 기초 + 예치금 충전 + 출고 포장/패킹리스트 기초
- 연결된 Phase: Phase 10
- 직전 Phase 결과: Phase 9에서 주문 묶음 -> 상품 라인 -> SKU 라인 구조와 SKU별 작업·정산 근거가 완료됐다.

## 1.1 플랫폼 독립 원칙 확인

- 고객 계약: 고객이 출고 전에 예치금 충전을 요청하고, 운영자 확인 뒤 잔액과 원장이 같은 사실로 남는다.
- 이번 Phase의 입력 방식 후보: 웹 충전 요청 폼, 운영자 승인/거절 폼
- 이번 Phase의 출력 방식 후보: 웹 대시보드, 예치금 관리, 운영자 확인 목록, 원장 내역
- 이번 Phase의 실행 엔진 후보: 무관
- 고객 계약에서 제외할 구현 선택지: 실제 계좌값 노출, PG/간편결제, 은행 API, 자동 입금 확인, 유료 API 실호출, 실DB 쓰기, 정식 운영 전환
- 입력 방식이 바뀌어도 유지되어야 하는 고객 결과: 충전 요청 금액, 상태, 운영자 확인 결과, 잔액, 원장이 일치한다.
- 출력 방식이 바뀌어도 유지되어야 하는 고객 결과: 셀러와 운영자가 같은 충전 상태를 본다.
- 실행 엔진이 바뀌어도 유지되어야 하는 고객 결과: 승인되지 않은 충전은 잔액에 반영되지 않고, 승인된 충전은 한 번만 반영된다.

## 2. Phase 고객 목표

- 이번 Phase에서 고객이 끝낼 수 있어야 하는 일: 셀러가 예치금 충전을 요청하고, 운영자 확인 뒤 잔액과 원장을 확인한다.
- 이 일이 안 되면 고객에게 생기는 피해: 출고 요청 전에 돈을 넣었는지, 잔액이 왜 늘었는지, 운영자가 확인했는지 알 수 없어 출고와 정산이 멈춘다.
- 이번 Phase가 끝나면 고객이 다음으로 할 수 있는 일: 충전된 예치금과 SKU별 출고 요청을 이어서 포장단위/패킹리스트 흐름으로 넘길 수 있다.

## 3. 고객 결과 요청

### 고객 결과 요청 1

- 고객에게 생기는 변화: 로그인한 셀러가 예치금 충전 요청을 남기고, 운영자는 그 요청을 승인하거나 거절해 잔액과 원장을 맞춘다.
- 완료 기준:
  - 셀러는 예치금 관리 화면에서 충전 금액을 입력해 충전 요청을 만들 수 있다.
  - 충전 요청에는 금액, 입금자명/확인 메모, 상태(`PENDING`, `APPROVED`, `REJECTED`), 요청 시각, 처리 시각, 처리자가 남는다.
  - 운영자는 충전 대기 목록에서 요청을 승인하거나 거절할 수 있다.
  - 승인하면 셀러 잔액이 요청 금액만큼 증가하고, 예치금 원장에 충전 기록이 1건 남는다.
  - 거절하면 잔액과 원장이 증가하지 않고, 셀러가 거절 상태와 사유를 본다.
  - 이미 승인/거절된 요청은 다시 처리되지 않는다.
  - 셀러는 자기 충전 요청과 원장만 본다. 다른 셀러 요청, 잔액, 원장은 보이지 않는다.
  - 화면은 실계좌·실입금·실결제가 아직 오픈 전 확정임을 명확히 보여준다.
- 이번에 안 하는 것: 실계좌 실제값 노출, PG/간편결제, 은행 API, 자동 입금 확인, 실제 입금 매칭, 세금계산서, 환불, 정식 운영 전환, 1688 외부 연동, 포장단위/패킹리스트.
- 고객 피해 방지 기준:
  - 운영자 승인 전 잔액이 증가하면 실패.
  - 승인됐는데 원장이 없거나, 원장은 있는데 잔액이 안 늘면 실패.
  - 같은 충전 요청이 두 번 승인되어 잔액이 중복 증가하면 실패.
  - 거절된 충전 요청이 잔액에 반영되면 실패.
  - 다른 셀러의 충전 요청이나 원장이 보이면 실패.
  - 실계좌·실입금·실결제가 실제 되는 것처럼 보이면 실패.
- 실제 한 건의 QA 예시:
  - 셀러 A의 현재 예치금 잔액은 0원이다.
  - 셀러 A가 100,000원 충전 요청을 만든다.
  - 요청 직후 잔액은 아직 0원이고, 상태는 `충전 대기`다.
  - 운영자가 해당 요청을 승인한다.
  - 셀러 A 예치금 잔액은 100,000원이 되고, 원장에는 `충전 승인 +100,000원` 1건이 남는다.
  - 대시보드 홈과 예치금 관리 화면은 같은 잔액을 보여준다.
  - 같은 요청을 다시 승인하려 하면 처리되지 않는다.
  - 셀러 B는 셀러 A의 충전 요청과 원장을 볼 수 없다.
- 통과해도 실패인 경우: 셀러 화면에는 잔액이 늘었는데 원장이 없거나, 운영자 화면에는 승인됐는데 셀러 화면은 대기 상태로 남는 경우.

## 4. PM 후보 판단

- PM이 판단하고 진행할 수 있는 것: 충전 요청 상태, 승인/거절 흐름, 잔액·원장 일치 기준, 로컬 기준판의 실입금 담장 문구.
- 대표님 판단이 필요한 것: 실제 입금 계좌, 자동 입금 확인 방식, PG/간편결제 도입 여부, 환불/세금계산서 정책, 정식 운영 전환.
- 후보가 너무 클 수 있는 이유: 실제 계좌, 은행 API, 결제사, 자동 입금 매칭, 환불까지 넣으면 외부 계약과 돈 정책이 필요해 Phase 10을 넘는다.
- 후보가 너무 작을 수 있는 이유: 충전 요청만 만들고 운영자 승인·잔액·원장이 연결되지 않으면 고객은 실제로 충전이 됐는지 확인할 수 없다.
- 같은 고객 약속 안에서 묶을 수 있는 인접 고객 결과: 대시보드 홈 잔액 반영, 예치금 원장 충전 내역, 운영자 승인/거절 목록.
- 절대 묶으면 안 되는 다른 고객 약속: 실계좌 실제값, PG, 은행 API, 자동 입금 확인, 환불 자동화, 포장단위, 패킹리스트, 실송장.
- Team Leader에게 요청하는 범위 판단: 이 후보가 Phase 10 한 Phase로 적정한지 `superpowers:brainstorming`으로 판단한다.

## 5. Team Leader 범위 판단 후 PM 반영

- Team Leader 판단: 적정
- PM 재조정 내용: 없음
- Phase 경계 확정 여부: 확정
- 대표님 재정렬 필요 여부: 현재 없음
