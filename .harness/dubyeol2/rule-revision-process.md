# 두별2 운영 규칙 개정 프로세스

두별2 운영 규칙은 문서 계약이다. 전체 교체가 아니라 개정으로 관리한다.

## 적용 대상

이 절차는 아래처럼 여러 프로젝트와 역할에 영향을 주는 규칙을 바꿀 때 적용한다.

- 두별2 사고법
- `AGENTS.md` 공통 규칙
- PM 역할 규칙
- Team Leader 규칙
- Builder 서브에이전트 규칙
- Phase gate, 완료 보고, 외부감리, 머지 승인 규칙
- Superpowers 연동 기준

단일 프로젝트의 고객 약속, 화면 문구, Phase 실행 결과, 제품별 예외는 이 절차가 아니라 해당 프로젝트의 `feature.md`, `phases.md`, `phase-packets/`, `changes.md`에 기록한다.

## 표준 절차

1. 현재 정본 확인
   - `AGENTS.md`, `.harness/dubyeol2/PROJECT.md`, `.harness/dubyeol2/current-run.md`를 먼저 읽는다.
   - 바꾸려는 규칙이 공통 규칙인지, 프로젝트별 예외인지 구분한다.

2. 개정안 작성
   - 기존 철학 중 유지되는 것, 추가되는 것, 삭제되는 것, 의미가 바뀌는 것을 분리한다.
   - 대표님이 고객, 돈, 업무, 범위, 만족 관점에서 승인할 수 있는 언어로 쓴다.

3. 대표님 승인
   - 정본 변경 전 대표님 승인을 받는다.
   - 승인 근거 문구와 날짜를 남긴다.

4. 기존 정본 archive
   - 바뀌기 전 중요한 정본은 `archive/`에 보존한다.
   - 파일명에는 규칙명, 이전 버전, 날짜를 넣는다.

5. 새 revision 작성
   - `revisions/`에 새 정본 승격 기록을 남긴다.
   - 유지, 추가, 삭제, 의미 변경, 승인 근거를 포함한다.

6. 중앙 정본 갱신
   - 중앙 하네스의 `AGENTS.md`와 관련 `.harness/dubyeol2/` 문서를 갱신한다.
   - 숨은 규칙을 만들지 않고 파일 계약으로 남긴다.

7. 하네스 버전 생성
   - 여러 프로젝트에 영향을 주는 변경이면 `VERSION.md`를 갱신한다.
   - 버전에는 포함된 정본, 승인 근거, 동기화 대상, 검증 상태를 적는다.

8. 설치 프로젝트 동기화
   - twostarhub, white7, 자비스처럼 두별2가 설치된 프로젝트에 같은 공통 규칙을 반영한다.
   - 프로젝트별 제품 맥락은 덮어쓰지 않는다.
   - 각 프로젝트에 적용된 하네스 버전을 `.harness/dubyeol2/VERSION.md`로 남긴다.

9. 프로젝트 검증
   - 각 프로젝트에서 `AGENTS.md`, `CLAUDE.md`, `.harness/dubyeol2/VERSION.md`가 같은 하네스 버전을 가리키는지 확인한다.
   - 필요한 경우 `changes.md`에 동기화 사실을 남긴다.

10. PM 실제 테스트
   - 새 규칙으로 PM이 고객 흐름 -> 기능 약속 -> Phase -> 고객 결과 요청을 한 번 dry-run 한다.
   - 문서는 맞지만 PM이 실제로 쓰기 어렵다면 정본 승격을 완료로 보지 않는다.

## 완료 기준

- 기존 정본이 archive에 보존되어 있다.
- 새 revision이 있다.
- `VERSION.md`가 현재 하네스 버전과 포함 규칙을 설명한다.
- 설치 프로젝트에 같은 버전이 기록되어 있다.
- PM 실제 테스트가 PASS 또는 보류 사유를 남겼다.
- `changes.md`에 변경 이유와 재발 방지 기준이 남았다.

## 금지

- 기존 정본을 기록 없이 전면 교체하지 않는다.
- 프로젝트별 제품 문서를 공통 하네스 버전업으로 덮어쓰지 않는다.
- 대표님 승인 없이 역할 경계, Phase gate, 금지선을 바꾸지 않는다.
- Superpowers 공식 workflow 내용을 두별2 문서에 복제하지 않는다. 두별2는 역할과 게이트만 고정한다.
