# feature.md

상태: 완료 (2026-07-06) — Feature 4 Phase 7 감리 PASS·로컬 main 머지

> Feature 3(두리무역 1:1 비공개 기준판)은 Phase 6 Machine Check PASS, Claude 교차감리 대기. Phase 6 감리 기준은 `phase-packets/phase-006/feature-contract-snapshot.md`에 보존한다. 이 문서는 **Feature 4: 로그인 후 셀러 출고 요청·예치금 흐름**을 닫는다.

## 1. 고객이 하려는 일 (L1)

- 로그인한 셀러가 견적 완료된 주문을 보고, 출고 요청을 하고, 예치금 차감과 출고 상태를 한 번에 확인한다.
- 이 기능이 끝나면 기존 `주문 접수 → 입고 확인 → 견적 확인` 흐름이 `출고 요청 → 예치금 정산 → 출고 상태 확인`으로 이어진다.
- 대표님 발화: "로그인후 셀러기능 구현하자," (2026-07-05)

## 2. 제품·사업 판단 (L2)

- 이번 기능은 돈 받고 서비스하는 핵심 흐름의 다음 단계다. 셀러가 견적을 보고도 출고/결제를 못 하면 서비스가 돈 흐름으로 이어지지 않는다.
- 이번에는 실계좌, 실입금, 실결제 연동을 하지 않는다. 로컬 기준판에서 예치금 잔액, 차감 내역, 주문 상태 변경이 고객 눈에 일관되게 보이는 것을 먼저 닫는다.
- 실제 입금확인 자동화, 결제사 연동, 송장·통관·택배 연동은 운영 정책과 외부 계약이 필요하므로 후속 Phase로 넘긴다.

## 3. 범위 (L3)

- **하는 것**: 견적 완료 주문 상세에서 출고 요청, 예치금 잔액 확인, 예치금 차감, 예치금 내역 기록, 주문 상태 변경, 출고관리 화면에서 상태 확인.
- **셀러 화면 기준**: 주문 상세, 예치금 관리, 출고관리, 대시보드 홈 상태 카드가 서로 같은 사실을 보여야 한다.
- **이번에 안 하는 것**: 실입금, 실계좌 노출, PG/간편결제, 외부 배송조회, 송장 발급, 통관 신고, 실제 창고 출고 처리, 반품/환불, 스마트오더/장바구니/내상품 고도화.
- 대표님 정렬: PM이 "견적 후 출고/예치금"을 첫 범위로 제안했고, 대표님이 "그래"로 잠금 (2026-07-05).

## 4. 절대 생기면 안 되는 일

- 예치금이 차감됐는데 주문 상태가 바뀌지 않는 것.
- 주문 상태만 바뀌고 예치금 내역이 남지 않는 것.
- 다른 셀러의 주문, 잔액, 예치금 내역이 보이는 것.
- 실계좌, 실입금, 실결제처럼 오해하게 만드는 것.
- 출고 요청이 실제 창고 출고 완료나 송장 발급처럼 보이는 것.
- 대표님 승인 없이 deploy, 원격 push, 원격 머지, 실서비스 변경, 유료 API 실호출, 실DB 쓰기, secret 출력 또는 커밋을 하는 것.

## 5. 합격 예시 (L4)

- 셀러가 견적 완료된 주문 1건을 열고 출고 요청을 누른다.
- 화면에서 예치금이 견적 금액만큼 차감되고, 예치금 내역에 차감 기록이 남고, 주문 상태가 출고 요청/출고 준비로 바뀐다.
- 같은 주문이 출고관리 화면에 나타나고, 대시보드 홈 상태 카드도 그 상태를 반영한다.
- 통과해도 실패: 돈만 차감되고 주문 상태가 안 바뀌거나, 상태만 바뀌고 예치금 내역이 없거나, 다른 셀러 주문/잔액이 보이는 경우.
- 대표님 승인: PM 합격 예시 제안에 "그래" (2026-07-05).

## 6. 담장 (L5)

- 로컬 개발 환경 기준으로만 구현한다.
- 예치금은 로컬 앱 안의 테스트 원장으로만 처리한다. 실계좌·실입금·실결제·외부 결제사·유료 API 호출 없음.
- 셀러가 실제 돈이 빠져나간 것으로 오해하지 않도록, 실운영 전까지 입금/결제 버튼은 테스트 기준 또는 준비 중 담장을 명확히 둔다.
- 송장번호, 통관, 국내택배 추적, 창고 실제 출고 완료는 이번 Phase에서 만들지 않는다.
- 기존 Phase 6 기준판 감리는 별도 대기 상태다. Phase 6 PASS 전에도 Mode B 규칙에 따라 Phase 7 준비·빌드는 가능하지만, Phase 6 완료·main 머지는 교차감리 PASS 후에만 한다.
