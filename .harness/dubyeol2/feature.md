# feature.md

상태: 대표님 승인 (2026-07-04)

이 문서는 이번 기능 약속, 즉 고객에게 약속할 변화를 적은 합의서다. 개발은 이 문서가 승인되기 전 시작하지 않는다.

> 제품 비전(전체 로드맵): 사업자(쿠팡·11번가·1688 셀러) 대상 중국→한국 통합 물류 플랫폼 — 구매대행 + 배송대행 + 검수 + LCL 콘솔 + 통관 + 추적 + 자동화. 두리무역(duly.co.kr) 수준 품질을 **순차 완성**으로 도달한다. 이 feature.md는 그 로드맵의 **첫 기능 약속 하나**만 닫는다.

## 0. 연결된 고객 흐름

- 전체 고객 흐름: 사업자 셀러가 중국에서 소싱 → 우리에게 주문/물건 맡김 → 중국 창고 입고·검수 → (다음) LCL 국제배송 → 통관 → 한국 도착 → 추적·정산
- 연결된 메뉴탭 / 화면 영역: 셀러 주문 접수 · 셀러 대시보드(입고·검수·견적)
- 이번 기능이 해결하는 막힘: 셀러가 중국 소싱 물건을 맡겼을 때 "내 물건이 안전하게 들어왔는지, 하자는 없는지, 총비용이 얼마인지"를 믿을 수 있게 확인할 방법이 없음
- 막힘이 남으면 고객에게 생기는 피해: 물건 상태·수량·비용을 모른 채 맡겨야 해서 분실·하자·숨은비용 리스크를 셀러가 떠안음. 신뢰가 안 생겨 물량을 못 맡김
- 이번 기능이 끝나면 고객이 다음으로 할 수 있는 일: 안심하고 검수 완료된 물량을 LCL 국제배송(다음 기능 약속)으로 넘김

## 0.1 플랫폼 독립 원칙 확인

- 고객 계약: 셀러가 주문을 맡기면 → 중국 창고에 안전하게 입고되고 → 입고 증거(입고 사진·외포장 이상 안내, 유료 신청 시 수량·외관·하자 검수)를 보고 → 항목별 투명 견적을 확인한다
- 입력 방식 후보: 웹 주문 폼, 상품 링크 붙여넣기, 엑셀 대량 업로드, (다음 Phase) 1688 API 연동
- 출력 방식 후보: 웹 대시보드, 알림(카카오/이메일)
- 실행 엔진 후보: 무관
- 고객 계약에서 제외할 구현 선택지: 특정 결제사, 특정 DB, 1688 OAuth, AI HS코드 엔진, 특정 WMS 제품
- 입력/출력/실행 엔진이 바뀌어도 유지되어야 하는 고객 결과: "안전 입고 + 검수 증거 + 투명 견적"이라는 세 가지 확인은 방식이 바뀌어도 유지된다

## 1. 고객이 하려는 일

- 고객: 중국에서 소싱하는 사업자 이커머스 셀러 (쿠팡·11번가·스마트스토어·1688 셀러)
- 고객이 끝내려는 일: 중국에서 산(또는 대신 사줄) 물건을 맡기고, 안전하게 창고에 들어왔는지·하자 없는지·총 얼마인지를 **믿을 수 있게 확인**하는 것
- 이 일이 안 되면 고객에게 생기는 문제: 깜깜이로 맡겨야 하니 분실·하자·숨은비용을 셀러가 떠안고, 그래서 물량을 못 맡김

## 2. 고객이 끝났다고 느끼는 순간

- 고객이 보는 결과: 셀러 대시보드에서 자기 주문 건의 **입고 완료 상태 + 기본 입고 확인(입고 사진 1~2장 · 외포장 이상 안내) + 항목별 견적(상품가·수수료·예상 국제운임)**을 한눈에 확인. 수량 확인·상세 검수(외관·하자)는 셀러가 요청한 건에만 유료 부가서비스로 표시. (두리무역식 — 대표님 확정 2026-07-04, 추후 컷팅 예정)
- 고객이 안심할 수 있는 증거: 실제 물건이 찍힌 입고 사진 1~2장, 외포장 이상 시 안내, (유료 요청 시) 수량·외관·하자 검수 결과, 실제 청구와 어긋나지 않는 견적
- 고객이 다음 행동으로 넘어갈 수 있는 조건: 검수·견적을 확인하고 "이 물량을 국제배송으로 보내겠다"를 결정할 수 있는 상태

## 3. 돈 받고 서비스할 때 절대 생기면 안 되는 일

- 입고 확인·검수 결과(입고 사진, 외포장 이상 안내, 유료 검수 시 수량·외관·하자)가 실제 물건과 다르게 표시되는 것 (셀러가 잘못 믿고 결정)
- 기본에 포함 안 되는 검수(수량·외관·하자)를 무료인 것처럼 오인하게 표시하는 것 (나중에 "왜 돈 받냐" 분쟁)
- 견적에 없던 비용이 나중에 추가 청구되는 것 (숨은비용)
- 셀러의 사업자 정보(사업자등록증·통관고유부호)·주문 데이터가 노출·유출되는 것
- 다른 셀러의 주문/물건이 섞여 보이는 것

## 4. 이번에 되는 것

- 셀러가 주문을 접수한다 (상품 링크/수량/옵션, 구매대행 또는 배송대행 구분)
- 중국 창고 입고를 등록하고 **기본 입고 확인(입고 사진 1~2장 촬영 · 외포장 이상 안내)**을 남긴다
- 수량 확인·상세 검수(외관·하자)는 셀러가 요청한 건에만 유료 부가서비스로 남긴다 (두리무역 기준: 검수 ¥35/시간 또는 ¥1/개, 추가 사진 ¥1/장, 동영상 ¥5/분 — 우리 요율은 대표님 확정 대기)
- 셀러 대시보드에서 자기 주문의 입고 상태·입고 사진·(유료 요청 시)검수 결과·항목별 투명 견적을 확인한다

## 5. 이번에 안 하는 것

빼도 이번 목표가 죽지 않는 것은 여기에 둔다. (전부 "안 함"이 아니라 **다음 기능 약속/Phase로 이어붙임**)

- LCL 국제배송·콘솔(혼적)·통관·한국 도착·배송 추적 → 두 번째 기능 약속(다음 Phase)
- 1688 OAuth 자동 연동, 주문 자동 불러오기 → 뒤 Phase(수동/붙여넣기로 시작)
- AI HS코드 추천, 3PL 풀필먼트, WMS 바코드/재고위치 → 뒤 Phase
- 실제 1688 매입 결제 자동화 → 이번 Phase 제외(대금·매입은 수동/모의)

## 6. 합격 예시

추상어가 아니라 실제 한 건으로 쓴다. **(대표님 리뷰 필요 — L4)**

- 예시 이름: 쿠팡셀러 "미니가전 셀러 A"의 100개 소싱 건
- 고객 입력: 1688 상품 링크 1건 + 수량 100 + 옵션(색상) + 구분(구매대행). 배송받을 중국 창고는 우리가 발급한 주소. 유료 검수(수량+외관)를 추가 신청.
- 고객이 보는 결과: 대시보드에 이 주문이 "입고 완료"로 뜨고, **기본 입고 확인(입고 사진 2장 · 외포장 이상 없음)**, 유료 검수를 신청한 이 건은 **검수 결과(실입고 수량 100/100 · 외관 정상 · 하자 0건)**도 함께, 그리고 견적: 상품가 + 구매대행 수수료 + 검수비 + 예상 국제운임(무게 12kg / 부피 0.08CBM 기준)이 **항목별로** 표시.
- 합격 기준: 셀러가 주문 접수부터 입고·검수·견적 확인까지 화면에서 스스로 완결한다. 입고 사진이 실제 물건이고, 유료 검수 결과(수량·외관·하자)가 실제와 일치하고, 견적 항목이 나중 실제 청구와 어긋나지 않는다.
- 통과해도 실패인 경우: 화면 흐름은 다 되는데 ① 입고 사진/검수 결과가 실제와 다르거나 ② 견적에 없던 비용이 나중에 붙거나 ③ 유료 검수를 신청 안 한 셀러가 수량 확인이 된 것으로 오인하는 화면이면 → **통과해도 실패**

## 7. 기능 전체 QA 시나리오

### 7.1 정상 흐름

- 시작 상태: 셀러가 회원(사업자)으로 로그인, 우리 중국 창고 주소를 발급받은 상태
- 고객 행동: 주문 접수(링크·수량·옵션·구분, 선택 시 유료 검수 신청) → 물건이 중국 창고 도착 → (운영) 입고 등록·입고 사진 1~2장·외포장 이상 기록·(신청 건) 검수 결과 기록 → 셀러가 대시보드에서 입고·검수·견적 확인
- 기대 결과: 셀러가 자기 주문 건의 입고 상태·입고 사진·(신청 시)검수 결과·항목별 견적을 정확히 본다
- 합격 기준: 6번 합격 예시와 동일

### 7.2 실패 흐름

- 시작 상태: 외포장 파손이 입고 시 발견됨, 또는 유료 검수 신청 건에서 수량 불일치(예: 100개 주문, 97개 도착)·하자 발견
- 고객 행동: 셀러가 대시보드 확인
- 기대 결과: 외포장 이상은 기본 안내로, 수량 불일치(97/100)·외관 이상·하자 내용은 신청한 검수 결과로 **명확히 표시**되고, 견적/후속 처리에 반영
- 고객 피해 방지 기준: 불일치·하자를 숨기지 않고 셀러가 결정 전에 반드시 보게 한다. 검수 미신청 건은 "수량 미확인"임이 화면에서 오해 없이 드러난다

### 7.3 돈 받고 서비스할 때 절대 실패하면 안 되는 흐름

- 시작 상태: 여러 셀러의 주문·물건이 창고에 동시에 존재
- 고객 행동: 셀러 A가 로그인해 자기 주문 확인
- 반드시 지켜야 하는 결과: 셀러 A는 자기 주문·검수·견적만 보고, 다른 셀러 것은 절대 안 보인다. 사업자 정보는 유출되지 않는다.
- 실패 시 고객 피해: 타 셀러 물건/정보 노출 → 신뢰 붕괴·개인정보/영업비밀 유출

## 8. UI QA 시나리오 요약

- UI 여부: 있음 (셀러 주문 접수 화면 + 셀러 대시보드)
- 고객이 화면에서 해야 하는 일: 주문 접수, 입고·검수·견적 확인
- 화면에서 절대 헷갈리면 안 되는 지점: 기본(입고 사진+외포장 안내)과 유료 검수(수량/외관/하자)의 경계 표시, 견적 항목별 금액(숨은비용 없음), 내 주문과 남의 주문 구분
- 외관 기준: 두리무역(duly.co.kr) 홈페이지 외관을 그대로 벤치마킹해 시작, 추후 대표님이 직접 편집 (대표님 지시 2026-07-04)
- 실제 UI QA 별도 의뢰 필요 여부: 필요 (디자인 기준은 `design-system.md` 정렬 후)

- PM이 정렬된 기준 안에서 판단할 수 있는 것: 첫 기능 약속 경계, 합격 예시 초안, QA 시나리오
- 대표님 판단이 필요한 업무/돈/범위 질문 (L5 담장 후보):
  - 구매대행 시 실제 1688 매입·결제는 이번 Phase에서 **모의/수동**으로 둔다 (유료 결제 실호출 = 금지선). 확인 필요.
  - 견적의 실제 요율은 **두리무역 공개 요율을 초기값으로 사용**(수수료 5%+VAT, 해운 첫 1kg ¥25+kg당 ¥5, 항공 서류비 ¥32.5+100g당 ¥1.8, LCL $65-70/CBM, 검수 ¥35/시간, 부가서비스 단가표 등 — 수집 원본: 세션 스크래치패드 `duly-rates-collected.md`, 벤치마크 정본으로 이관 예정). 대표님이 추후 컷팅/조정.
  - 기본 무료 = 입고 사진 1~2장 + 외포장 이상 안내 (두리무역식). 수량·외관·하자 검수는 유료 옵션. 대표님 확정(2026-07-04), 추후 컷팅 예정.
  - 실서비스 DB 쓰기·배포·원격 push는 대표님 명시 승인 전까지 안 함 (금지선).
  - 셀러 사업자 정보(사업자등록증·통관고유부호)는 민감정보로 취급, 노출·로그·커밋 금지.
