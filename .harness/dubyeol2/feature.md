상태: 완료 (2026-07-06) — Feature 5 Phase 9~11 완료, Phase 8 보류

> 이전 완료: Feature 3(두리무역 1:1 비공개 기준판) Phase 6 감리 PASS·로컬 main 머지, Feature 4(견적 후 출고 요청·예치금 차감) Phase 7 감리 PASS·로컬 main 머지, Feature 5 Phase 9~11 감리 PASS·로컬 main 반영. 대표님 지시로 기존 Phase 8 후보(우리 회사용 정식 운영 전환)는 보류한다.

## 1. 고객이 하려는 일 (L1)

- 로그인한 셀러와 운영자가 상품별·SKU별 수량, 입고, 검수, 견적, 정산을 섞이지 않게 볼 수 있게 한다.
- 이 기능이 끝나면 기존 `주문 1건 = 상품/옵션 텍스트 한 줄` 구조가 `주문 묶음 -> 상품 라인 -> SKU 라인` 구조로 바뀌고, 이후 예치금·출고·포장·패킹리스트를 같은 기준 위에 올릴 수 있다.
- 대표님 발화: "8은 일단 보류, 9-11까지 먼저 진행해" / "지금은 상품마다, sku마다 구분되어있어? 구분되어있어야 작업이랑 정산을 하디 않을까?" / "포장단위랑 패킹리스트는 출고요청때 붙이면 어때? 처음부터는 필요할까?" / "그래" (2026-07-06)

## 2. 제품·사업 판단 (L2)

- Phase 8(우리 회사 정보·약관·요율 컷팅·정식 오픈 전환)은 보류한다. 따라서 이번 기능은 고객 공개용 정식 운영 전환이 아니라 비공개/로컬 기준판의 기능 확장이다.
- SKU는 작업·정산의 최소 단위다. 상품명/옵션 텍스트만 있으면 부분 입고, 부분 하자, SKU별 단가, SKU별 검수, SKU별 정산을 믿을 수 없다.
- 포장단위와 패킹리스트는 처음 주문 구조에 넣지 않고, 출고요청/출고처리 Phase에서 붙인다. SKU는 "무엇이 몇 개냐"이고 포장은 "어떻게 나가냐"라서 실제 창고 작업 후 달라질 수 있기 때문이다.
- 예치금 충전은 SKU 구조가 깔린 뒤 진행한다. 그래야 이후 차감 내역과 출고비가 어떤 SKU/주문 묶음에서 생겼는지 연결된다.

## 3. 범위 (L3)

- **Phase 8 보류**: 우리 회사용 정식 운영 전환(실주소, 전화번호, 계좌, 사업자정보, 약관 본문, 요율 컷팅, 외관 추가 편집)은 지금 하지 않는다.
- **Phase 9 — SKU 라인 구조**: 주문 묶음, 상품 라인, SKU 라인, SKU별 수량/옵션/단가/입고수량/검수결과/하자수량/견적 근거를 분리한다.
- **Phase 10 — 예치금 충전 신청/확인**: 셀러 충전 요청, 충전 금액 입력/프리셋, 입금자명/연락처, 충전 대기 목록, 운영자 승인/거절, 잔액 반영, 예치금 원장 기록.
- **Phase 11 — 출고요청 + 포장단위 마커 + 패킹리스트 기초**: 출고요청 이후 BOX-1, BOX-2 같은 포장단위 마커를 붙이고, 박스별 SKU 구성·수량·무게·CBM을 기록해 패킹리스트를 만들 수 있게 한다.
- **이번에 안 하는 것**: Phase 9에서 포장단위 확정, 패킹리스트 생성, 실송장 발급, 정식 고객 공개, 실계좌 실제값 노출, PG/간편결제, 은행 API, 외부 배송조회 API, 세관 API, 1688 OAuth/API/크롤링, 자동 구매, 자동 송장 동기화, 반품 환불 자동화, deploy, 원격 push.

## 4. 절대 생기면 안 되는 일

- 상품 A 빨강 50개와 상품 A 파랑 50개가 한 줄로 뭉쳐 보여서 입고·검수·견적·정산이 섞이는 것.
- 일부 입고, 일부 하자, 일부 누락이 주문 전체 상태 하나로 덮이는 것.
- SKU별 단가가 다른데 주문 총액만 보여 고객이 정산 근거를 확인할 수 없는 것.
- Phase 9에서 포장단위가 확정된 것처럼 보여 실제 출고 작업과 충돌하는 것.
- 다른 셀러의 SKU 라인, 입고수량, 검수결과, 견적 근거가 보이는 것.
- 1688 검색·구매·운송장 자동 연동이 실제 되는 것처럼 보이는 것.
- Phase 8을 보류했는데도 실주소·실전화·실계좌·사업자정보·약관 본문을 확정값처럼 넣는 것.

## 5. 합격 예시 (L4)

- 셀러가 한 주문 묶음에 상품 A 빨강 50개, 상품 A 파랑 50개, 상품 B L 30개를 넣는다.
- 운영자가 입고 처리할 때 빨강 50개 입고완료, 파랑 45개 입고/5개 부족, 상품 B L 30개 중 2개 하자를 SKU별로 기록한다.
- 견적에는 SKU별 상품가·수량·검수비·중국배송비 배분 근거가 보이고, 주문 묶음 합계도 같이 보인다.
- 포장단위와 패킹리스트는 아직 생성하지 않고, "출고요청 때 포장단위 배정"으로 남는다.
- 통과해도 실패: SKU별 입고·검수·견적은 보이지만 정산은 주문 전체 한 줄로만 남거나, 포장단위가 Phase 9에서 확정된 것처럼 보이는 경우.

## 6. 담장 (L5)

- 로컬 개발 환경 기준으로만 구현한다.
- Phase 9는 SKU 구조까지만 한다. 포장단위 마커, 박스별 무게/CBM, 패킹리스트 생성은 Phase 11 출고요청/출고처리에서 한다.
- Phase 10 예치금 충전은 로컬 테스트 원장 + 운영자 수동 승인으로만 처리한다. 실입금 확인 자동화, 은행 API, PG, 간편결제 없음.
- Phase 8 보류 상태이므로 실주소, 실전화, 실계좌, 사업자정보, 약관 본문은 확정값으로 넣지 않는다.
- Phase 11 배송·통관·국내택배 추적은 운영자 수동 입력만 한다. 외부 배송조회·세관 API 호출 없음.
- 1688 상품 검색, 상품 상세, 구매 자동화, 운송장 자동 동기화는 이번 기능에서 제외한다.
- 대표님 승인 없이 deploy, 원격 push, 원격 머지, 실서비스 변경, 유료 API 실호출, 실DB 쓰기, secret 출력 또는 커밋을 하지 않는다.
