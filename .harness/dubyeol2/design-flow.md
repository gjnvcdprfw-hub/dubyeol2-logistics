# design-flow.md — 두별2 디자인 트랙

상태: 두별2 디자인 트랙 절차 정본

Claude Design(claude.ai/design)을 두별2의 디자인 작업 경로로 쓴다. 디자인 확정은 대표님이 캔버스에서 보면서 하고, 구현은 두별2 빌드 게이트를 그대로 탄다.

전제: 프로젝트의 디자인 시스템 프로젝트(예: twostarhub = "ALFA Design System")가 claude.ai에 있고, 코드의 공유 부품이 `/design-sync`로 올라가 있다. 없으면 먼저 `/design-sync`로 만든다.

## 세 경로

1. **신규 기능(화면 포함)**: 두별2 정렬(L1~L5) → 디자인 브리프 → 캔버스 확정 → Export 핸드오프 → 빌드. 계약 없이 캔버스부터 가지 않는다(고객 계약 → 출력 방식 순서).
2. **감 잡기**: 정렬 도중 캔버스 러프 목업을 L4 합격 예시의 증거로 쓴다. 러프는 초안이지 확정이 아니다. 확정은 정렬이 닫힌 뒤다.
3. **순수 디자인 수정(기능 불변, 발화 ⑦)**: 정렬 풀코스 없이 캔버스 수정 → 핸드오프 → `design-system.md` 기준 갱신 → Builder 반영.

## 디자인 브리프 — PM이 정렬 닫힘 직후 생성

- 이 화면으로 고객이 끝내는 일 (L1)
- 지켜야 할 업무규칙·절대 생기면 안 되는 것 (L2)
- 이번에 안 넣는 것 (L3)
- 합격 장면 (L4) — "이렇게 보이면 합격" 실제 한 건
- 지시 한 줄: "이 프로젝트의 Design System 부품만 사용"

전달: 대표님이 Claude Design 입력창에 붙여넣는다(+손그림·스크린샷 첨부). PM이 DesignSync로 디자인 프로젝트 안에 브리프 파일을 넣어둘 수도 있다.

## 대표님 캔버스 조작 (3단계)

1. Claude Design 열기 → Design system이 이 프로젝트 것인지 확인
2. 브리프 붙여넣기 → 캔버스에서 말로 반복 수정 ("버튼 위로", "이 표는 빼")
3. 확정되면 **Export → "Send to local coding agent"**

## 핸드오프 수신 (빌드 측)

- Export 번들은 **확정 디자인 기준**이다. 게이트를 우회하지 않는다 — 팀리더가 `02-plan.md`에 번들을 연결하고, Builder가 그 부품·화면대로 구현하며, Codex 외부감리가 디자인 시스템 위반을 본다.
- 번들이 기존 `design-system.md` 기준과 어긋나면 대표님 확정이 새 기준이다 — `design-system.md`를 갱신하고 `changes.md`에 기록한다.
- 화면 확정본의 원본은 디자인 프로젝트(screens/)다. 레포에는 구현 코드만 둔다. 같은 것을 두 곳에 보관하지 않는다.

## 동기화 유지

- 공유 부품 코드가 바뀌어 로컬 main에 머지되면, 기능 완료 시점에 `/design-sync`를 재실행해 캔버스 부품을 최신으로 맞춘다.
- 동기화는 부품 단위 증분으로 한다. 통째 교체 금지.

## 재구축 경로

디자인 시스템을 다시 만드는 두 경우다. (캔버스 프로젝트 정비·리셋은 재구축이 아니라 유지보수로 본다.)

### A. 리디자인 — 기존 프로젝트의 룩을 갈아엎기

1. 정렬부터. 전 화면에 영향 가는 결정이므로 L1~L5를 풀코스로 닫는다 — 특히 L2(취향·브랜드 방향)와 L4("이렇게 보이면 합격"인 실제 화면 한 장).
2. 캔버스 탐색·확정: 새 디자인 프로젝트(또는 자유 모드)에서 대표님이 보면서 확정한다. 기존 부품에 얽매이지 않는다.
3. 빌드: `design-system.md`를 새 기준으로 갱신하고 `changes.md`에 기록 → 공유 부품을 새 스타일로 개편(팀리더→빌더, 기능 단위 배치) → 화면 반영.
4. 재동기화: 새 부품이 로컬 main에 머지되면 `/design-sync`로 캔버스 부품을 교체한다.

### C. 신규 구축 — 디자인 시스템이 없는 프로젝트

1. 코드에 공유 부품(컴포넌트 라이브러리)이 있는지 확인한다. 없으면 부품 승격이 첫 빌드 작업이다.
2. `/design-sync`로 디자인 프로젝트를 생성·업로드한다. GitHub 레포·디자인 파일에서 임포트하는 방법도 있다.
3. `design-system.md`에 기준(색·글꼴·간격·부품 목록)을 정본화한다. DESIGN.md 규격을 따른다 — 확정 값은 YAML 토큰, 이유는 산문, `npx -y @google/design.md lint`로 검증.
4. 이후는 위의 세 경로를 그대로 따른다.

## 담장

- Lazyweb은 대표님이 명시 요청할 때만 쓴다(기존 규칙 그대로).
- 디자인 확정 권한은 대표님이다(L2 취향 판단). PM·팀리더가 캔버스 확정을 대행하지 않는다.
