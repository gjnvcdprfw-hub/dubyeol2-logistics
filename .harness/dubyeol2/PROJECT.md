# PROJECT.md

상태: 두별2 v2.3 프로젝트 대문 템플릿

이 문서는 프로젝트의 오래가는 기준이다. 새 기능마다 갈아엎지 않는다. 고객에게 해주는 일, 고객 흐름, 메뉴탭/화면 영역, 절대 깨지면 안 되는 기준을 기록한다. Phase별 실행 증거는 `phase-packets/`에 둔다.

두별2의 운영 목표는 대표님이 기술자가 아니어도 고객, 돈, 업무, 범위, 만족만 판단하면 제품이 완성되게 하는 것이다.

두별2의 핵심 정본 트리는 이 문서 안에 둔다.

```text
프로젝트 목표
└── 고객 흐름
    └── 메뉴탭 / 화면 영역
        └── 기능 약속
            └── Phase
                └── 고객 결과 요청
```

`feature.md`, `phases.md`, `phase-packets/`, `approvals.md`, `changes.md`는 이 트리를 보조하거나 증거를 남기는 문서다. 고객 목표, 고객 흐름, 메뉴탭, 기능 약속, Phase, 고객 결과 요청의 현재 정본은 이 문서에서 끊기지 않게 관리한다.

## 플랫폼 독립 원칙

고객 계약은 특정 기술, 입력 방식, 출력 방식, 실행 엔진에 의존하지 않는다.

두별2는 항상 고객 계약 -> 입력 방식 -> 출력 방식 -> 실행 엔진 순서로 판단한다. 입력 방식, 출력 방식, 실행 엔진은 구현 선택지이며 고객 계약에 포함하지 않는다.

- 고객 계약: 사업자 셀러가 중국 소싱 물건을 맡기면 → 안전하게 입고되고 → 입고 증거(사진·외포장 안내, 유료 신청 시 검수)를 보고 → 항목별 투명 견적을 확인하고 → (다음 기능) 국제배송·통관·한국 도착까지 추적한다
- 입력 방식 후보: 웹 주문 폼, 상품 링크 붙여넣기, 엑셀 대량 업로드, (뒤 Phase) 1688 API 연동
- 출력 방식 후보: 웹 대시보드, 알림(카카오/이메일)
- 실행 엔진 후보: 무관 (Codex/Claude/GPT 등 구현 선택지)
- 입력 방식이 바뀌어도 유지되어야 하는 고객 계약: "안전 입고 + 입고 증거 + 투명 견적" 확인
- 출력 방식이 바뀌어도 유지되어야 하는 고객 계약: 동일
- 실행 엔진이 바뀌어도 유지되어야 하는 고객 계약: 동일
- 고객 계약에 넣으면 안 되는 구현 선택지: 특정 결제사, 특정 DB, 1688 OAuth, AI HS코드 엔진, 특정 WMS 제품

## 두별2 역할 경계

PM은 기능 약속을 닫은 뒤 Phase를 최종 확정하지 않는다. PM은 고객 약속, 제외 범위, QA 예시가 담긴 후보 고객 결과 요청을 만들고 Team Leader에게 넘긴다.

Team Leader는 `superpowers:brainstorming`으로 후보 고객 결과 요청의 범위가 한 Phase로 진행하기에 너무 크거나 작은지 판단한다. 너무 크면 PM에게 Phase 분리를 요청하고, 너무 작으면 같은 고객 약속 안에서 묶을 인접 고객 결과를 PM에게 요청한다.

PM은 Team Leader의 범위 판단을 받은 뒤 고객 약속, 제외 범위, 만족 기준이 바뀌지 않는 선에서 Phase 경계를 확정한다. 기준이 바뀌면 대표님과 다시 정렬한다.

Team Leader는 Superpowers 공식 workflow를 따른다. 구현은 `superpowers:subagent-driven-development`로 Builder 서브에이전트가 맡는다.

두별2는 Team Leader의 세부 판단을 다시 묻지 않는다. Superpowers 스킬을 지켰는지와 PM에게 되돌릴 고객 약속 변경이 있는지만 확인한다.

## 1. 프로젝트 한 줄 설명

- 고객에게 해주는 일: 중국 소싱(1688 등) 물건의 구매·입고·검수·견적·국제배송·통관을 한 곳에서 투명하게 처리해주는 중국→한국 통합 물류 플랫폼 (벤치마크: 두리무역 duly.co.kr)
- 주요 고객: 중국에서 소싱하는 사업자 이커머스 셀러 (쿠팡·11번가·스마트스토어·1688 셀러)
- 고객이 이 프로젝트로 끝내려는 일: 깜깜이 없이(분실·하자·숨은비용 리스크 없이) 중국 물건을 한국 판매처까지 가져오는 일

## 2. 주요 고객 흐름

| 흐름 | 고객이 하려는 일 | 끝났다고 느끼는 순간 | 깨지면 안 되는 기준 |
|---|---|---|---|
| 1 | 주문을 맡기고 입고·검수·견적을 확인 | 대시보드에서 입고 증거와 항목별 견적을 보고 "보내자" 결정 가능 | 입고 증거·견적이 실제와 일치, 숨은비용 없음 |
| 2 | 검수 완료 물량을 LCL 국제배송·통관으로 한국까지 | 한국 도착·수령 확인 | 분실·오배송 없음, 통관 문제 사전 안내 |
| 3 | 전 과정 추적·정산 | 건별 비용·상태를 언제든 확인 | 내 데이터만 보임, 청구가 견적과 일치 |

## 3. 고객 흐름과 메뉴탭

메뉴탭은 고객 흐름을 처리하는 장소다. 메뉴탭 진행률은 참고용이고, 진짜 완료 기준은 고객 흐름이 끝났는지다.

| 고객 흐름 | 메뉴탭 / 화면 영역 | 고객이 하려는 일 | 현재 상태 | 연결 기능 약속 | 연결 Phase |
|---|---|---|---|---|---|
| 1 | 셀러 주문 접수 | 링크·수량·옵션으로 주문 맡기기 | 준비 전 | Feature 1 | Phase 1 |
| 1 | 셀러 대시보드(입고·검수·견적) | 입고 상태·증거·견적 확인 | 준비 전 | Feature 1 | Phase 1 |
| 2 | (다음) 출고·배송·통관 | 국제배송 신청·추적 | 준비 전 | Feature 2(예정) | 미정 |

## 4. 돈 받고 서비스할 때 절대 생기면 안 되는 일

- 입고 확인·검수 결과(입고 사진, 외포장 안내, 유료 검수 시 수량·외관·하자)가 실제 물건과 다르게 표시되는 것
- 견적에 없던 비용이 나중에 추가 청구되는 것 (숨은비용)
- 셀러의 사업자 정보·주문 데이터 노출·유출, 다른 셀러의 주문/물건이 섞여 보이는 것
- 기본에 포함 안 되는 검수를 무료인 것처럼 오인하게 표시하는 것

## 8. 기능 약속별 Phase와 고객 결과 요청

Phase 번호는 프로젝트 전체 누적 번호다. 기능이 바뀌어도 Phase 1로 리셋하지 않는다. Task 번호는 각 Phase 안에서 `Task 1`부터 다시 시작한다.

### Feature 1: 주문 접수 → 입고·기본확인 → 투명 견적

- 고객 약속: 셀러가 주문을 맡기면 안전 입고 + 입고 증거(사진 1~2장·외포장 안내, 유료 신청 시 수량·외관·하자 검수) + 항목별 투명 견적을 확인한다
- 연결 고객 흐름: 흐름 1
- 연결 메뉴탭: 셀러 주문 접수 · 셀러 대시보드(입고·검수·견적)
- 승인 상태: 승인
- 승인일: 2026-07-04
- 승인 근거: "승인할게 두별2에 맞게 진행해" (approvals.md §1)

Feature 1은 Team Leader 범위 판단(너무 큼) 후 Phase 1~3으로 세로 분할했다 (phases.md §2). Phase 2 = 입고 증거 확인, Phase 3 = 항목별 견적 확인(Feature 1 완료).

## 9. 운영 메모

- 기존 프로젝트에 두별2를 처음 적용하는 경우, 먼저 `handover.md`를 채운다.
- 이미 두별2가 적용된 프로젝트라면 이 문서와 최근 `changes.md`, `approvals.md`를 먼저 읽는다.
- Team Leader는 같은 메인 Codex가 PM에서 전환한 역할이며, 서브에이전트로 만들지 않는다.
- PM과 Team Leader는 기획, 설계, 판단, 배정만 하고 직접 구현하지 않는다. 구현은 `superpowers:subagent-driven-development`로 Builder 서브에이전트가 맡는다.
- PM은 후보 고객 결과 요청을 만들고, Team Leader가 `superpowers:brainstorming`으로 범위를 판단한 뒤 Phase 경계를 확정한다.
- 고객 결과 요청은 개발 티켓이 아니라 고객 결과 단위 작업이다. Builder Task는 Team Leader가 `superpowers:writing-plans` 후 만든다.
- 새 Phase부터는 `phase-packets/phase-n/`의 5개 파일로 단계 진입 여부를 판단한다.
- Phase 완료 후 PM의 전체 보고 없이는 Phase close, 머지 승인 요청, 다음 Phase 진행으로 넘어가지 않는다.
- 오래 걸림만으로 실패 판단하지 않는다. 실패는 실패 증거로 판단한다.
